但是，“存在 CLI 开关”不等于“实验完全可控”。当前最重要的问题是：各 runner 默认基线不统一、BOA runner 可静默叠加 normalization、run id 未编码全部关键参数、旧 RHL-SE checkpoints 缺少 training config、P0 的 Batch32 只是 eval batch、多个分支同时修改 `trainer/trainer.py`。这些问题会造成实验可运行但结果无法准确归因。

## 1. 严重度排序的问题

### [高] 1.1 Runner 默认值不统一，跨实验表存在隐藏混杂

当前主要默认值：

| Runner | Buffer | Batch | norm | base path | 用途 |
|---|---:|---:|---|---|---|
| 当前分支 `run.sh` | 8192 | 32 | none | 空/外部传入 | 通用入口 |
| `run_rhl_norm.sh` | 8196 | 64 | l2_sqrt | 20260606 step0 | normalization |
| `tools/run_boa_rhl.sh` | 8196 | 32（不足则16） | none | 20260606 step0 | BOA |
| `tools/run_rhl_se_val_driven.sh` | 消费既有 checkpoint | eval batch 32/16 | 继承成员 | 成员路径 | ensemble eval |

风险：把 `run_rhl_norm.sh`、BOA 和 P0 结果放在一张表时，Buffer、训练 Batch、base path 与 norm 同时不同。特别是 P0 的“Batch32”是评估 loader Batch Size，不能称为 Batch32 训练结果。

立即可执行的修改方案：

1. 定义统一 neutral baseline：Batch32、Buffer8196、gamma1、legacy init、legacy scale、norm none、固定 base checkpoint。
2. 所有 runner 从同一个只读 baseline config 或公共 shell defaults 读取，但每个 runner 明确打印最终参数。
3. summary 增加 `train_batch_size` 与 `eval_batch_size` 两列，禁止共用 `batch_size`。
4. 对历史结果增加 provenance 标签，不重写旧日志。

验证实验：使用每个 runner 的 neutral/off 配置各跑一个 seed，要求 all/old/new 在浮点容差内一致。

### [高] 1.2 BOA 与 normalization 可静默融合，run id 又未记录组合

`tools/run_boa_rhl.sh` 允许通过 `RHL_NORM` 覆盖归一化，但输出路径主要编码 init/scale/seed/Batch，没有完整编码 Buffer、gamma、norm 和 global seed。由此可能出现两个不同配置写入难以区分的目录，或把 BOA+norm 误报为 BOA。

立即可执行的修改方案：

```text
RUN_ID=boa_${init}_${scale}_buf${buffer}_g${gamma}_norm${norm}_gseed${seed}_rseed${rhl_seed}_bs${batch}
```

并增加：

```text
ALLOW_COMBINED_RHL=0
```

当 BOA 非 baseline 且 norm 非 none 时，若没有显式设为 1 则退出并打印配置。该 guard 不是禁止组合，而是要求组合有明确实验身份。

### [高] 1.3 RHL-SE 成员缺少强制配置兼容性检查

旧 seed checkpoints 仅含模型对象/状态，缺少完整 `training_config`。当前 ensemble 主要按 CLI 重建数据集并加载 pickled model，没有系统比较成员的 base checkpoint、Buffer、gamma、norm、训练 Batch、init/scale 和任务协议。

风险：不同训练配置的 checkpoint 可以被无提示混入同一 ensemble。性能变化可能来自成员异构，而不是 class-wise/gate 机制。

立即可执行的修改方案：

1. 新增 member manifest，逐个写入 checkpoint sha256 和所有关键训练参数。
2. `eval_rhl_ensemble.py` 默认执行同构校验；不一致时列出字段并退出。
3. 只有显式 `--allow_heterogeneous_members` 才允许继续，并在输出 JSON 标记。
4. 对旧 checkpoint 生成外部 sidecar metadata，不修改 checkpoint 文件本身。

### [高] 1.4 三条 feature branch 都修改 Trainer，直接合并会产生语义冲突

分支概况：

| 分支 | 主要内容 | 与 `main` 关系 | 共享热点 |
|---|---|---|---|
| `feature/rhl-se-boa-p0-p1` | RHL seed、RHL-SE、BOA、config 修复 | 基于较早 main，4 个独有提交 | `trainer/trainer.py`、parser/ckpt |
| `feature/deeplabv3plus-control` | DeepLabV3+ 对照、resume runner | 更早分叉 | `trainer/trainer.py`、run.sh |
| `feature/resume-step0-epoch39` | step0 恢复训练 | 更早分叉 | `trainer/trainer.py` |

当前 `main` 还有该 RHL feature 未包含的 `run.sh/run_origin.sh` 路径更新。简单依次 merge 可能在语法上通过，但覆盖 base checkpoint、resume 或 RHL config 语义。

立即可执行的集成方案：

1. 先在独立 integration branch 合入最新 main。
2. 按“checkpoint/path 基础设施 -> resume -> model architecture -> RHL feature”顺序人工整合 Trainer。
3. 使用 `git range-diff` 和逐分支 tests 验证，不以“冲突已解决”为完成标准。
4. DeepLab worktree 当前 `run_v3plus.sh` 有未提交用户改动，合并前先由用户决定是否提交；不得自动覆盖。

### [中] 1.5 `run_rhl_norm.sh` 的默认值不是 neutral off

该 runner 默认 `RHL_NORM=l2_sqrt`。作为方法 runner 可以接受，但作为复现/对照入口容易误开已证明不占优的模式。

修改方案：默认 `none`，具体实验必须显式 `RHL_NORM=power` 或旧模式；或者保留默认但脚本启动时输出醒目的完整配置和 `METHOD_ON=1`。更推荐前者，因为 off-path 应是统一基线。

### [中] 1.6 `gaussian` 参数名与真实初始化不一致

当前 `gaussian` legacy path 实际使用 `nn.Linear.reset_parameters()`，并非 Gaussian。代码功能没有因此错误，但日志和论文解释会错误归因。

修改方案：

- 保留 `gaussian` 作为兼容 alias；
- 新 canonical 名称用 `legacy_uniform`；
- parser/日志打印 deprecation notice；
- 单元测试验证 alias 输出逐元素一致；
- 历史结果仍标注原 CLI 名，分析中写明真实分布。

### [中] 1.7 Trainer 临时修改共享 opts，未来组合方法容易泄漏状态

现有流程在构造 step0/step1 数据或模型时临时修改 `opts.curr_step` 再恢复。当前修复已补恢复和 config，但共享可变对象仍较脆弱；PGH prototype loader、Snapshot base checkpoint 和多 runner 组合会增加遗漏恢复的可能。

修改方案：使用 `copy.copy/deepcopy` 或 dataclass-style 独立 config，为 step0、step1、prototype、eval 分别构造不可共享的局部配置。增加测试：构建各 loader 后原始 opts 不变。

### [中] 1.8 缺少统一的 batch-limit smoke 参数

当前没有一致的 `--max-train-batches/--max-val-batches/--max-test-batches`。任何 Trainer、Dataset、prototype 或 weighted RLS 修改都难以执行低成本真实链路 smoke test。

修改方案：三个参数默认 `-1`，在 loop 边界统一截断，并写入 run summary；只用于 smoke 的 run id 必须标记 `smoke`，不得进入正式结果汇总。

### [中] 1.9 DeepLabV3+ 报告状态落后于真实实验

DeepLab feature worktree 中已有结果：all mIoU 约 `68.9663`、old `78.1453`、new `39.5933`，新类明显弱于当前 RHL 强基线，但已有报告仍写“待完成”。

修改方案：在 DeepLab 分支单独更新其报告，记录 checkpoint、日志、命令、Batch、是否 resume 和结果来源。不要在当前 RHL 文档提交中顺手修改该分支。

### [低] 1.10 伪标签开关在当前 VOC 15-5 sequential step1 实际不生效

代码存在 `--use_pseudo_label`、`--pseudo_label_confidence`，但当前 Trainer 条件要求 `curr_step > 1` 且 `setting != sequential`。因此它不能被列为“当前 VOC 15-5 step1 可消融方法”，也没有专用 runner。

修改/验证方案：先明确伪标签目标协议，再新增条件单元测试和专用 runner。若设计就是只服务非 sequential 后续 step，应在 parser help 和报告中明确，而不是改变逻辑。

### [低] 1.11 `utils/tasks.py` 存在未抛出的 `NotImplementedError`

不支持 dataset 的分支写了 `NotImplementedError` 表达式但没有 `raise`，可能继续执行并在更远处以不清晰错误失败。

修改方案：改为 `raise NotImplementedError(...)` 并增加非法 dataset 测试。本报告按用户要求不直接修改。

### [低] 1.12 checkpoint 使用完整 pickle model 加载

当前 checkpoint 含完整模型对象并由 `torch.load` 读取。对本地可信产物可用，但不应加载来源不明的 checkpoint。

修改方案：长期迁移为 architecture config + state_dict；短期在文档和 loader 中明确 trusted-local-only，并校验路径/hash。

## 3. 现有代码的耦合关系

### 3.1 已明确且可控的耦合

#### RHL init x scale x gamma

随机矩阵初始化和 scale 会改变 hidden feature 范数，gamma 又决定 Ridge 正则。三者不是独立超参。当前代码允许分别设置，但实验设计必须增加 scale-matched gamma 对照。

#### Buffer x antithetic

antithetic 的总维度与独立方向数不同。`BUFFER=8196` 下 antithetic 只有 4098 独立方向，因此 BOA-2 的退化不能只归因为“反向成对无效”。需要同总维度和同独立方向两类实验。

#### norm x BOA

norm 可以覆盖 BOA 引起的幅值差异，也可能把 BOA 的有效机制消掉。代码上可组合，但当前 runner 身份不清楚。应使用显式组合 guard 和 2x2 实验。

### 3.2 尚未明确、会影响归因的耦合

#### 训练 Batch Size x C-RLS 更新轨迹

解析更新按 batch 顺序累积，Batch32/64 不应被视为纯吞吐参数。已有 normalization Batch32 结果不能与 Batch64 结果直接归因于 norm。必须固定训练 Batch，并区分 eval batch。

#### global seed x RHL seed

虽然 `rhl_seed` 已隔离随机矩阵，但 global seed 仍影响数据顺序、augmentation 和其他随机过程。seed 复核应固定 global seed，只改变 RHL seed；稳定性实验再在外层改变 global seed。

#### base checkpoint x method

当前不同 runner 对 step0 base path 的默认值不同。若 base checkpoint 不同，所有 step1 差异都不再是纯方法差异。manifest 必须记录 hash，runner 启动前打印并校验。

#### ensemble member config x fusion method

如果成员来自不同 Buffer、Batch、norm 或 BOA 配置，ensemble gain 同时包含“成员更强”和“融合更好”。同构与异构 ensemble 必须分表。

#### branch integration x behavior

DeepLab/resume/RHL 三个分支对 Trainer 的改动互相覆盖时，可能出现 checkpoint 加载顺序、config 恢复或 model architecture 的隐性变化。需要 integration branch 上按功能链路 smoke，而不是只跑 py_compile。

## 4. 可立即执行的耦合验证实验

### 4.1 Baseline runner 等价性

目的：证明不同 runner 的方法 off-path 一致。

```text
run.sh neutral
run_rhl_norm.sh RHL_NORM=none
run_boa_rhl.sh INIT=legacy SCALE=legacy RHL_NORM=none
```

固定 base checkpoint/hash、global/RHL seed、Batch32、Buffer8196、gamma1。比较 checkpoint config、feature stats 和 all/old/new。

### 4.2 BOA x norm 2x2

| | norm none | PowerNorm/旧 norm on |
|---|---|---|
| legacy | baseline | norm only |
| orthogonal | BOA only | BOA+norm |

目的：检测 normalization 是否覆盖或放大 BOA 的几何收益。

### 4.3 Training Batch x eval Batch

训练 Batch32/64 各自产生 checkpoint；每个 checkpoint 再用 eval Batch16/32 评估。若同 checkpoint 的 eval batch 不变而训练 batch 改变结果，则差异属于训练更新轨迹。

### 4.4 同构 x 异构 RHL-SE

1. 同构：仅 RHL seed 不同；
2. 异构：plain、BOA、PGH 或 Snapshot 成员；
3. 分别比较 uniform/class-wise/RHL-SE2。

目的：区分 gate 能力与成员质量/多样性。

### 4.5 分支集成回归

集成 branch 上逐项执行：

- baseline neutral；
- resume step0 单 epoch；
- DeepLab forward + step1 smoke；
- RHL norm off/on；
- BOA baseline/orthogonal；
- checkpoint save/load round-trip。

每项保存 config diff，不允许用最后一个综合 run 替代单功能回归。
