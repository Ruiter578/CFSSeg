## 2. 为什么 step1 batch size 是 32，不是 64

结论：当前用 32 是为了协议一致、代码默认一致和实验可比；不能把 step1 batch size 64 当作自然默认值。

代码事实：

1. `run.sh` 默认：

```bash
DEFAULT_BATCH_SIZE="${DEFAULT_BATCH_SIZE:-32}"
SPECIAL_BATCH_SIZE="${SPECIAL_BATCH_SIZE:-32}"
```

2. `tools/run_adaptive_pseudo_label.sh` 默认：

```bash
BATCH_SIZE="${BATCH_SIZE:-32}"
STEP0_BATCH_SIZE="${STEP0_BATCH_SIZE:-32}"
```

3. `datasets/init_dataset.py` 直接用 `opts.batch_size` 构造 train loader：

```python
DataLoader(..., batch_size=opts.batch_size, drop_last=True)
```

4. `trainer/trainer.py` 里：

```python
self.total_itrs = self.opts.train_epoch * len(self.train_loader)
```

所以 batch size 会改变 train loader 长度、每个 epoch 的 iteration 数、drop_last 丢弃样本边界，以及 batch 里的样本组成。

对伪标签尤其重要的是：

- `batch_global` 的阈值来自当前 batch 内所有候选像素的 quantile；
- `batch_class` 的阈值来自当前 batch 内每个旧类候选像素的 quantile；
- 因此 batch size 从 32 改成 64，不只是显存变大，也会改变 batch 内候选分布和阈值估计稳定性。

对于 `fixed` 和 `artifact_class`，batch size 的语义影响小一些，但它仍然会改变 dataloader iteration 数和丢弃样本边界。为了让 Phase A / Phase B 的结论只比较阈值策略，而不是混入 batch size 因素，当前统一使用 32 是合理的。

如果后续要研究 batch size 64，应把它作为独立消融：

```text
同一模型
同一 step0 checkpoint
同一 pseudo-label strategy
同一 seed
只改 batch size 32 -> 64
所有对照重跑
```

否则不能把 64 的结果和现有 32 的 fixed0.6 / batch_class q0.1 / artifact_class 直接放在同一主表里比较。

## 3. V3+ 应该什么时候接入自适应伪标签线

结论：当前分支已经包含 V3+ 模型代码，不需要再“把 V3+ 加入 feature 分支”。推荐实验路线是先把 adaptive-pseudo-label 在 DeepLabV3 上做通、做完、审清楚，再合并到 main；之后在 main 上做 V3+ 验证。若时间紧或 GPU 空闲，也可以在当前 feature 分支上提前跑 V3+ exploratory validation，但最终主线结论应以合并后的 main 或明确 commit 为准。

当前代码事实：

当前 `feature/adaptive-pseudo-label` 已经能搜索到：

```text
network/modeling.py: deeplabv3plus_resnet101
network/_deeplab.py: DeepLabHeadV3Plus
network/_deeplab.py: aspp_up
trainer/trainer.py: forward_air_features
utils/parser.py: air_feature_source choices include aspp_up
```

也就是说，V3+ 不是缺失代码，而是尚未作为 adaptive pseudo-label 的正式实验条件运行。

推荐路线：

1. 当前阶段先完成 DeepLabV3 adaptive pseudo-label：

```text
PhaseB artifact calibration
artifact_class q0.05 / q0.10 / q0.20 step1 训练
summary 与 fixed0.6 / batch_class q0.1 对比
必要时做 raw-mask audit 或降级为消融
```

2. 代码和结果稳定后，把 `feature/adaptive-pseudo-label` 合并到 `main`。

3. 在 `main` 上做 V3+ 验证：

```text
MODEL=deeplabv3plus_resnet101
AIR_FEATURE_SOURCE=auto
使用 V3+ 对应 step0 BASE_SUBPATH
重新跑同样的 adaptive pseudo-label 最优策略
```

为什么不建议一开始就切 V3+：

- 当前 adaptive pseudo-label 的机制有效性还没有超过 `fixed0.6` 强 baseline；
- V3+ 会引入 backbone/head/feature source 变量；
- 如果 adaptive 在 V3+ 上涨或跌，很难判断是阈值机制、teacher calibration、AIR feature source，还是 V3+ 本身导致；
- 论文叙事上应先证明方法在原 CFSSeg/DeepLabV3 协议下成立，再说明它能迁移到 V3+。

允许的例外：

如果 PhaseB 的 DeepLabV3 结果已经明确超过或接近 `fixed0.6`，且 GPU 空闲，可以直接在当前 feature 分支上提前跑一组 V3+ exploratory validation。此时必须标注：

```text
branch = feature/adaptive-pseudo-label
commit = <exact commit>
status = exploratory before main merge
```

正式报告或论文表格中，建议等代码合入 main 后再用 main commit 重跑或至少复核。

## 5. 项目级 `.codex` 体系如何触发和维护

结论：`AGENTS.md` 是最稳定的自动规则来源；skills 和子 agent 不是无条件自动运行，最好显式触发。

### 5.1 自动生效部分

在 `/root/2TStorage/lyc/SegACIL` 下工作时，项目级：

```text
AGENTS.md
```

会作为项目规则参与约束任务。它适合放：

- 长期协议边界；
- 代码质量底线；
- 实验启动规范；
- 服务器/GPU/Git 工作流；
- 当前主线技术事实；
- agent 必须避免的低级错误。

本次已经把 PhaseB artifact 路线的执行链和 raw-mask audit 边界同步进 `AGENTS.md`。

### 5.2 skills 如何手动触发

最稳方式是在请求开头显式写：

```text
[$using-superpowers](...)
[$ask-matt](...)
[$code-review](...)
[$ara-rigor-reviewer](...)
```

常用建议：

| 任务 | 建议 skill |
| --- | --- |
| 需要确认该走什么流程 | `ask-matt` |
| 每轮开始统一检查 skill 规则 | `using-superpowers` |
| 严格审代码 | `code-review` |
| 审论文/实验结论严谨性 | `ara-rigor-reviewer` |
| 构思新机制 | `brainstorming` / `grill-me` |
| 交接长会话 | `handoff` 类流程 |

### 5.3 子 agent 是否自动触发

不应默认自动触发。当前项目级规则建议：

```text
只在用户明确要求使用 subagents / 并行 agent / delegate 时使用。
```

原因是本项目很多文件涉及同一训练链路，多个 agent 同时改同一文件容易互相覆盖。更推荐的方式是：

- 主 agent 负责最终事实判断；
- 子 agent 只做可隔离的审查或检索；
- 不让多个子 agent 同时写同一文件。

### 5.4 是否每次任务后都更新 `.codex`

可以，但要克制。

应该更新：

- 新增长期有效的实验入口；
- 新增必须遵守的协议边界；
- 发现高频错误模式；
- 某个机制成为项目主线；
- 服务器/GPU/Git 工作流发生长期变化。

不应该更新：

- 某次实验的临时结果；
- 某个 checkpoint 的偶然指标；
- 尚未验证的假设；
- 大段日志；
- 可以放在 `AI_docs/idea验证与结论` 或 `AI_docs/代码改动报告` 的流水账。

推荐分工：

```text
AGENTS.md：长期规则和操作底线
Codex_Plans：下一步 agent 可执行计划
AI_docs/代码改动报告：代码改动和验证证据
AI_docs/idea验证与结论：实验结果和方法判断
AI_docs/idea构思与实验设计：方案设计和工作流
```

## 6. 长会话压缩会不会丢信息

结论：会丢失逐字历史和部分细节，但通常保留主线摘要。不能把自动压缩当成完整实验记录。

压缩后通常能保留：

- 当前目标；
- 主要文件路径；
- 大致完成状态；
- 关键结论；
- 明确的下一步。

压缩后可能丢失：

- 某条命令的完整输出；
- 某个中间判断的理由；
- 用户早期的措辞和偏好；
- 未写入文档的分支细节；
- 哪些改动是用户已有、哪些是 agent 新增。

因此，重要事项必须落盘：

1. 实验结论写入 `AI_docs/idea验证与结论`。
2. 执行计划写入 `Codex_Plans`。
3. 代码改动写入 `AI_docs/代码改动报告`。
4. 运行参数进入 `run_manifest.json`。
5. checkpoint / log / artifact 路径写入报告。
6. 关键规则沉淀到 `AGENTS.md`。

如果一个会话很长，建议在阶段切换时显式生成 handoff 文档，而不是依赖自动 compact。
