## 5. 第一性原理：buffer 为什么会影响结果

### 5.1 当前 step1 实际在做什么

固定 step0 权重后，step1 不是普通 SGD 训练分类头。代码路径是：

```text
输入图像
  -> step0 DeepLab backbone / classifier 抽 dense feature
  -> RandomBuffer / RHL 固定随机投影
  -> RecursiveLinear 闭式/递推求解分类器
  -> 测试 old/new/all mIoU
```

关键代码：

- `trainer/trainer.py` 中 `AIR.__init__()` 创建 `RandomBuffer` 与 `RecursiveLinear`。
- `AIR.feature_expansion()` 调用 `backbone.forward_air_features()`，再调用 `self.buffer(X)`。
- `AIR.fit()` 把 RHL 后的高维特征交给 `RecursiveLinear.fit()`。
- `RecursiveLinear.fit()` 更新 `R` 和 `weight`，核心是带正则的最小二乘。

### 5.2 数学上 buffer 是什么

每个像素先有一个 256 维特征向量：

```text
x in R^256
```

`RandomBuffer` 用一个固定随机矩阵把它扩展到高维：

```text
z = ReLU(W_random x)
z in R^D
```

这里的 `D` 就是 `buffer`。

然后 `RecursiveLinear` 求解：

```text
min_A ||Y - ZA||^2 + gamma * ||A||^2
```

闭式解依赖：

```text
Z^T Z + gamma I
```

所以 buffer 的第一性原理作用是：改变随机特征空间维度 `D`，从而改变 `Z` 的表达能力、`Z^T Z` 的谱结构、数值条件和正则项相对强度。

### 5.3 为什么不是越大越好

更大的 buffer 带来更多随机特征，相当于更多随机基函数。理论上表达能力更强，但有代价：

1. `Z^T Z` 是 `D x D` 级别矩阵，`D` 越大，计算和显存越重。
2. 随机特征不是学出来的，存在抽样方差；不同 `rhl_seed` 会产生不同随机子空间。
3. `gamma` 固定时，`D` 变化会改变特征总能量和正则项的相对比例。
4. 当前任务是像素级类别不均衡场景，new 类很容易被 old/background 压制；更大的随机空间可能更好分离 new 类，也可能放大噪声。

这解释了为什么 `8208`、`8224` 强于 `8192`，但 `8216` 又突然回落。

### 5.4 为什么 8224 的提升主要来自 new 类

`8224` 对 `8208` 的 old 类均值略低：

```text
old: 0.784277 vs 0.784566
```

但 new 类均值明显更高：

```text
new: 0.425188 vs 0.421462
```

这说明 `8224` 并不是“全面提升所有类别”，而更像是改善了新类的可分性。在 class-incremental segmentation 里，这很有价值，因为 step1 的核心困难通常就是新类学习和旧类保持之间的张力。

## 6. gamma 的补充观察

用户后来要求固定 `gamma=1`，所以正式主结论只基于 `gamma=1`。

历史上有少量 `gamma=0.1/1/10` 结果，集中在 `8196` 和 `8204`：

| buffer | gamma | all mIoU | old 0-15 mIoU | new 16-20 mIoU |
|---:|---:|---:|---:|---:|
| 8196 | 0.1 | 0.695983 | 0.782854 | 0.417993 |
| 8196 | 1 | 0.695983 | 0.782855 | 0.417992 |
| 8196 | 10 | 0.695982 | 0.782855 | 0.417987 |
| 8204 | 0.1 | 0.696493 | 0.784532 | 0.414770 |
| 8204 | 1 | 0.696493 | 0.784532 | 0.414769 |
| 8204 | 10 | 0.696492 | 0.784533 | 0.414763 |

这组结果说明：在 `rhl_norm=none`、`rhl_seed=-1`、当前 buffer 区间的这两个点上，`gamma` 从 `0.1` 到 `10` 几乎不改变结果。第一性原理解释是：当前 `Z^T Z` 的主导尺度可能远大于这些 gamma 差异，或者数据/特征结构让 ridge 项在这个范围里不是限制因素。

因此，当前不建议优先继续做纯 gamma 搜索。gamma 更适合在以下场景重新打开：

1. 启用 `rhl_norm=l2` 或 `l2_sqrt` 后，特征能量尺度变化。
2. buffer 扩展到更大维度后，`Z^T Z` 数值条件变化。
3. 出现 `torch.inverse`、NaN、cusolver 等数值稳定问题。

## 7. Shell 脚本中值得搜索的超参数

这里按“第一性原理影响力”排序，而不是按脚本里出现的顺序排序。

修订说明：`RHL_NORM` 曾经作为方法性改动做过实验，但由于涨点效果不佳已经被搁置。为了让当前 buffer 超参数搜索保持干净，本轮不再把 `RHL_NORM` 作为搜索轴；它应固定为 `none`。与 RHL 相关的 `rhl_seed` 也不作为“调参找最优”的方法轴，而只作为随机投影子空间的稳定性检查。

### 7.1 第一优先级：直接改变 AIR/RHL 特征矩阵的参数

| 参数 | 所在脚本 | 当前值 | 是否值得搜 | 原因 | 推荐下一步 |
|---|---|---:|---|---|---|
| `BUFFER` | `run.sh`, `run_trs.sh`, `run_rhl_norm.sh` | 8208/8192/8196 等 | 值得 | 直接改变 `Z` 的维度和 `Z^T Z` | 继续右侧扩展 `8228,8232,8236,8240` |
| `RHL_SEED` | `run.sh`, `run_rhl_norm.sh` | -1/1/2/3 | 只做稳定性检查，不作为默认值优化 | 改变随机投影子空间，用来估计随机特征抽样方差 | 对候选 buffer 补 `4,5` |
| `RHL_NORM` | `run.sh`, `run_rhl_norm.sh` | none | 本轮不搜，固定为 none | 这是已做过且效果不佳的搁置方法，属于方法消融，不应混入干净超参搜索 | 保持 `RHL_NORM=none` |
| `AIR_FEATURE_SOURCE` | `run.sh` | auto -> decoder | 值得，但要小心 | 改变 DeepLab 哪一层特征进入 AIR | 当前 `deeplabv3_resnet101` 支持 `decoder/aspp`，可做小规模对照 |
| `GAMMA` | `run.sh`, `run_trs.sh`, `run_rhl_norm.sh` | 1 | 本轮固定为 1 | 已有小样本显示 `none` 下几乎不敏感，且用户要求固定 | 不进入本轮队列 |

### 7.2 第二优先级：改变数据矩阵采样或数值路径的参数

| 参数 | 当前值 | 是否值得搜 | 判断 |
|---|---:|---|---|
| `DEFAULT_BATCH_SIZE` | 32 | 不建议作为性能调参，建议作为数值一致性检查 | step1 是递推闭式更新，batch size 理论上不该是“方法性能”来源，但有限精度和 batch 顺序可能导致微小差异。可用 `16/32/64` 做一次 sanity check，不要当主贡献。 |
| `RANDOM_SEED` | 1 | 值得做鲁棒性，不宜用于挑最优 | 它会同时影响 DataLoader shuffle、数据增强、全局 RNG、以及 `rhl_seed=-1` 路径，变量混杂。正式结论应做 robustness，不应 cherry-pick。 |
| `CROP_SIZE` | 513 | 不建议先搜 | 改变训练/验证输入尺度和数据增强语义，成本和可比性问题较大。 |
| `OUTPUT_STRIDE` | 8 | 不建议只在 step1 改 | step0 checkpoint 是按当前结构训练/保存的；如果要搜，应该成套重训 step0，再比较 step1。 |

### 7.3 当前固定 step0、只跑 step1 时不值得搜的参数

这些参数在 shell 脚本里存在，但对“固定 step0 + curr_step=1 的 AIR 闭式 fit”不是主要优化轴：

| 参数 | 原因 |

## 8. 建议的后续实验路线

### 8.1 最建议立即做：buffer 右侧扩展

原因：当前最强候选 `8224` 是搜索上边界。第一性原理上，边界最优意味着区间没有封闭。

建议：

```text
buffer = 8228, 8232, 8236, 8240
gamma = 1
rhl_norm = none
rhl_seed = 2
BASE_SUBPATH = 20260621_baseline_bs16_16_trs
```

为什么先用 `rhl_seed=2`：当前最佳单次是 `8224/rhl2`，用同一个 RHL seed 可以快速判断右侧是否继续上升。若右侧有点接近或超过 `8224/rhl2=0.699814`，再对该点补 `rhl_seed=1,3,4,5`。

### 8.2 第二建议：补 seed 稳定性

如果目标不是冲更高上限，而是稳妥确认默认值，建议补：

```text
buffer = 8208, 8224
gamma = 1
rhl_seed = 4, 5
```

这能回答 `8224` 的高均值是否依赖少数幸运随机子空间。

### 8.3 本轮不做：RHL_NORM 搜索

`RHL_NORM` 是 RHL 输出归一化方法，不是当前要继续混入的干净超参。它之前已经做过并因涨点效果不佳搁置，因此本轮明确不跑：

```text
RHL_NORM = l2_sqrt / layernorm / l2
```

当前所有新实验固定：

```text
RHL_NORM = none
GAMMA = 1
```

如果未来要重启 RHL normalization，应单独开一个“方法消融”实验计划，不与 buffer 搜索混排。

### 8.4 暂缓：AIR_FEATURE_SOURCE 小规模对照

当前 `deeplabv3_resnet101` 的 `auto` 会解析为 `decoder`。该模型还支持 `aspp`。

这也是改变进入 AIR 的特征来源，已经接近方法/架构消融，不适合混入本轮干净 buffer 搜索。可作为后续独立问题处理：

```text
AIR_FEATURE_SOURCE = decoder, aspp
buffer = 当前最佳候选
gamma = 1
rhl_seed = 2
```

但本轮不启动。

## 9. 不建议的搜索方向

1. 不建议继续在当前 `rhl_norm=none` 下大规模搜 `gamma`。已有结果显示 `0.1/1/10` 几乎无差。
2. 不建议把 `LR/LR_POLICY/TRAIN_EPOCH` 当成本轮 step1 搜索变量。它们主要属于 step0 常规训练。
3. 不建议只改 `OUTPUT_STRIDE` 跑 step1。要改就应重训匹配的 step0。
4. 不建议用 `random_seed` 挑最好结果。它应作为鲁棒性报告，而不是默认值选择依据。
5. 不建议把 `rhl_seed=2` 直接写成方法默认。它是本地最好随机子空间，不是可解释的通用默认。正式报告应给多 seed 均值/方差。

## 10. 对 run_trs.sh 默认值的建议

当前 `run_trs.sh` 中：

```bash
BUFFER="${BUFFER:-8208}"
GAMMA="${GAMMA:-1}"
RHL_NORM="${RHL_NORM:-none}"
```

如果目标是“保守复现当前单 seed Phase A 最强点”，保留 `BUFFER=8208` 是合理的。

如果目标是“按多 RHL seed 后的本地最强候选继续推进”，可以把默认候选改为：

```bash
BUFFER="${BUFFER:-8224}"
GAMMA="${GAMMA:-1}"
RHL_NORM="${RHL_NORM:-none}"
```

但我不建议现在把文档表述成“8224 是最终最优”。更严谨的工程路线是：

1. 先跑 `8228/8232/8236/8240` 的右侧扩展；
2. 再补 top buffer 的 `rhl_seed=4/5`；
3. 最后决定 `run_trs.sh` 默认 `BUFFER` 是 `8208`、`8224`，还是右侧新点。

## 11. 最终判断

当前可以给出的强结论：

> `8192` 是论文默认和合理基线，但不是当前本地最优；在当前固定条件下，`8208-8224` 区间整体优于 `8192`，其中 `8224` 在多 RHL seed 后表现最好，尤其提升 new 类 mIoU。

当前不能给出的强结论：

> 不能说 `8224` 是全局最优 buffer；不能说应该把论文默认 `8192` 普遍改成 `8224`；也不能用 `rhl_seed=2` 的单次最好结果代表方法真实性能。

下一步最有价值的实验不是继续泛泛扩大超参表，而是一个短而干净的验证：

```text
固定 gamma=1、rhl_norm=none、base checkpoint 不变；
用 rhl_seed=2 向右扫 buffer=8228/8232/8236/8240；
若右侧出现更优或接近 8224 的点，再补 rhl_seed=1/3/4/5。
```
