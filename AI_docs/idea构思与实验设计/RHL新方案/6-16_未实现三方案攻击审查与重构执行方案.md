## 0. 先给结论

当前不应再把所有后续工作都写成“RHL 多随机种子集成的自然延伸”。RHL-SE 已经证明：

```text
只改变 RHL seed
可以带来轻微互补
但成员预测高度相似
全局或手工 class-wise 融合难以稳定转化为明显 mIoU 增益
```

因此，后续三个方案需要重新定位：

| 优先级 | 方案 | 新定位 | 执行方式 |
|---|---|---|---|
| P0 | RHL-SE val-driven 收尾 | 固化已经验证的弱正向集成模块，停止 test 手工调参 | 立即收尾，并作为后续对照 |
| P1 | BOA-RHL | 改造 RHL 随机特征质量，优先验证单模型是否涨点 | 与 P0 并行推进，不等待 P0 失败 |
| P2 | PGH-RHL-lite | 在 RHL 外加入类别语义锚点，贡献更强但工程风险更高 | 根据 BOA/RHL-SE 结果决定是否继续 |
| P3 | step0 Snapshot + analytic head | 系统级集成上界和结项支撑，不是 RHL 机制创新 | 作为系统补充，不抢 RHL 主线 |

一句话路线：

```text
P0 固化 RHL-SE val-driven 收尾
P1 同步实现并启动 BOA-RHL single
P2/PGH 和 P3/Snapshot 根据证据补充，而不是把 P0/P1 做成单一路径依赖
```

2026-06-17 执行修订：本文件不再把方案推进写成“先做一个保守子集、失败后才允许转向”的防御性规划。P0 和 P1 本身服务于不同问题：P0 收敛推理集成结论，P1 改造 RHL 特征构造。二者可以在同一轮代码实现和实验启动中同时推进。

## 1. 从 RHL-SE 失败经验提炼出的审查标准

RHL-SE 的问题不是代码错，而是原始规划有三个隐含乐观假设。

### 1.1 不能把“理论合理”直接等价为“指标会上升”

RHL-SE 的理论依据是随机子空间集成可以降低 Monte Carlo 方差。但真实实验显示，不同 RHL seed 的预测分歧很小，pairwise disagreement 约为 `0.87%`，只有新类区域分歧更高。也就是说，方差确实存在，但太小。

后续方案必须先回答：

```text
它改变的是误差来源中的哪一项？
这个误差来源在当前结果里是否足够大？
它能否改变单模型预测，而不是只改变推理平均？
```

### 1.2 不能只在 test 上手工调权重

RHL-SE 后期出现了手工扫 test 权重的问题。它可以帮助诊断，但不能作为可靠方法结论。

后续所有需要选择参数的实验必须遵守：

```text
val split 用于选超参数
test split 只做最终一次或少量确认评估
文档必须记录 val 选择规则
```

### 1.3 不要把“符合结项集成系统”误当成“论文主创新”

结项指标要求集成学习与集成系统，但论文更需要可解释且有效的方法贡献。RHL-SE 和 Snapshot 更适合作为系统模块或辅助实验；BOA-RHL / PGH-RHL 更适合作为论文中的 RHL 机制升级。

## 2. 方案 B：step0 Snapshot + 解析增量头

### 2.1 原始动机

原始方案 B 来自经典 Snapshot Ensemble：

```text
step0 DeepLab 训练过程中保存多个 checkpoint
每个 checkpoint 作为一个 frozen feature extractor
分别执行 step1 RHL + C-RLS 解析增量
推理时融合多个完整 CFSSeg 模型的预测
```

它比 RHL-SE 更像传统集成系统，因为差异来自不同 step0 backbone checkpoint，而不是同一个 backbone 上的不同 RHL 随机投影。

### 2.2 攻击审查

#### 问题 1：它不是 RHL 机制改进

Snapshot 的主要差异来源是 step0 DeepLab 的参数，而不是 RHL。它会改变 frozen dense feature：

```text
image -> different DeepLab snapshots -> different feature maps -> different RHL heads
```

所以它不能被写成“改进 RHL 随机隐藏层”。更准确的定位是：

```text
CFSSeg analytic incremental learner 上的 system-level ensemble
```

这对结项指标有用，但对“RHL 机制创新”帮助有限。

#### 问题 2：last-k snapshot 可能高度相似

如果 step0 已经收敛，epoch 30/40/50 的模型可能非常接近。这样会复现 RHL-SE 的问题：

```text
成员看似不同
预测实际高度相关
推理平均只能带来很小提升
```

因此不能默认 last-k snapshot 有效。必须先做 snapshot 间差异诊断。

#### 问题 3：早期 snapshot 可能拖累旧类基础能力

如果使用较早 checkpoint，模型多样性会上升，但单模型质量可能下降。VOC 15-5 的 all mIoU 中旧类占 16/21，旧类下降会迅速抵消新类收益。

#### 问题 4：训练和存储成本高于 RHL-SE

每个 snapshot 都要执行一次 step1 realignment + incremental update，并在推理时保留完整模型。它不再是低成本 RHL head 集成。

### 2.3 重构后的方案

Snapshot 不建议立刻做完整 cyclic learning rate。先做“现有或低成本 checkpoint 的上界验证”。

#### 阶段 B0：先不训练，检查现有 step0 checkpoint 是否可用

执行目标：

```text
找到已有 step0 checkpoint
统计 checkpoint epoch 或保存时间
比较 step0 val/test 指标
比较 step1 后单模型指标
```

如果没有足够多已有 snapshot，再考虑重训 step0。

#### 阶段 B1：增加正式 `--base_ckpt`，避免复制覆盖 final.pth

不建议使用临时复制：

```bash
cp snapshot_epoch_40.pth step0/final.pth
```

原因是容易污染 baseline 目录。正式实现应新增：

```text
--base_ckpt checkpoints/.../step0/snapshot_epoch_40.pth
```

step1 加载逻辑：

```text
if curr_step > 0 and base_ckpt is not None:
    load base_ckpt as previous step feature extractor
else:
    follow original step{curr_step-1}/final.pth path
```

#### 阶段 B2：只选 3 个 snapshot，必须用 val 筛选

候选选择规则：

```text
候选 snapshot 至少间隔 5 到 10 epoch
每个 snapshot 跑 step1
先在 val 上记录单模型 all/new/old mIoU
只保留质量不低于 baseline 0.3 all mIoU 以内的成员
```

如果成员质量明显低于 baseline，不进入 ensemble。

#### 阶段 B3：test 只做三类评估

```text
best snapshot single
top-3 snapshot equal ensemble
val-selected weighted ensemble
```

如果 top-3 equal 没有超过 best single 至少 `+0.10 all mIoU`，则说明 snapshot 成员互补不足，停止扩展。

### 2.4 完整工作流和优先级

| 步骤 | 动作 | 产物 | 是否必须 |
|---|---|---|---|
| B0 | 搜索已有 step0 checkpoint/snapshot | checkpoint 清单 | 必须 |
| B1 | 实现 `--base_ckpt` | 可复用训练入口 | 必须 |
| B2 | 3 个 snapshot 分别跑 step1 | 3 个完整模型 | 可选，放在 BOA 后 |
| B3 | val 选成员和权重 | `snapshot_weights.json` | 必须 |
| B4 | test 一次性评估 | 系统级集成结果 | 必须 |

优先级：P3。它适合做结项“集成系统”补充，不应该抢 BOA-RHL / PGH-RHL 的主线时间。

### 2.5 预期结果

合理预期：

```text
all mIoU: +0.0 到 +0.4
new mIoU: 可能 +0.2 到 +0.8
old mIoU: 有下降风险
```

只有当 Snapshot 在 val 上表现出明显成员互补时，才值得继续。

## 3. 方案二：BOA-RHL

### 3.1 原始动机

原始 RHL 使用普通 Gaussian random projection：

```text
E = ReLU(F W^T)
```

其中 `W` 随机初始化并固定。BOA-RHL 的核心动机是：

```text
相同 buffer_size 下，让随机方向覆盖更均匀
同时用 antithetic pair 减少 ReLU 单侧截断
```

也就是：

```text
Gaussian W
-> block-orthogonal W
-> [W, -W]
-> ReLU([F W^T, -F W^T])
```

### 3.2 攻击审查

#### 问题 1：antithetic pair 会让独立方向数减半

如果 buffer 仍为 8196，`[W, -W]` 意味着只有约 4098 个独立方向。它补回了正负响应，但减少了方向数量。

所以 BOA-RHL 不是无条件更强，它在交换两件事：

```text
方向数量减少
方向覆盖和符号信息增强
```

必须通过实验判断哪一项更重要。

#### 问题 2：orthogonal scale 可能比初始化形式更关键

此前 RHL 归一化失败说明：只改尺度通常不会改变闭式解的核心能力。但是 BOA-RHL 如果 scale 设错，会直接让 C-RLS 的有效正则强度变化：

```text
E scale 变化
=> E^T E 量级变化
=> gamma 的相对强度变化
=> 解析头解发生偏移
```

所以 BOA-RHL 不能只测一个 `sqrt(2/C)`。至少需要做 scale audit。

#### 问题 3：block orthogonal 在 rows >> cols 时不是真正全局正交

DeepLab dense feature 通道通常为 256，而 buffer 为 8196。最多只能在每个 256 行块内正交，不可能让 8196 个方向全局互相正交。

因此文档表述不能夸大为“所有 RHL 维度正交”。准确说法应是：

```text
分块正交采样，使每个 block 内方向覆盖更均匀
```

#### 问题 4：BOA-RHL 可能提升稳定性但不提升均值

BOA 的效果可能是降低 seed 方差，而不是提高 best mIoU。如果只跑一个 seed，可能误判。

### 3.3 重构后的方案

BOA-RHL 应先做单模型最小闭环，不要一开始叠加 ensemble。

#### 参数设计

新增参数：

```text
--rhl_init gaussian
--rhl_init orthogonal
--rhl_init orthogonal_antithetic
--rhl_scale_mode legacy
--rhl_scale_mode kaiming
--rhl_scale_mode unit
```

解释：

| 参数 | 含义 |
|---|---|
| `gaussian` | 原始 RandomBuffer 初始化，作为 baseline |
| `orthogonal` | 分块正交，但不加反向成对 |
| `orthogonal_antithetic` | 分块正交加 `[W, -W]` |
| `legacy` | 尽量复现原始 weight 分布量级 |
| `kaiming` | `sqrt(2 / in_features)`，适合 ReLU |
| `unit` | 行向量单位范数，用于排查 scale 问题 |

#### 训练矩阵

第一轮只跑 4 个实验：

| 编号 | `rhl_init` | `rhl_scale_mode` | 目的 |
|---|---|---|---|
| BOA-0 | `gaussian` | `legacy` | 复现 baseline 路径 |
| BOA-1 | `orthogonal` | `legacy` | 分离正交采样效果 |
| BOA-2 | `orthogonal_antithetic` | `legacy` | 分离反向成对效果 |
| BOA-3 | `orthogonal_antithetic` | `kaiming` | 检查 ReLU scale 是否关键 |

第一轮不跑 K=3 ensemble。原因是需要先证明单模型特征构造有效。

#### 诊断指标

除了 mIoU，还要输出：

```text
RHL feature mean/std
RHL feature sparsity, 即 ReLU 后 0 的比例
E^T E trace 或平均对角值
old/new mIoU
per-class new IoU
```

如果 BOA 提升很小，但 sparsity 和 feature correlation 明显改善，说明方向正确但还需要调 gamma 或 scale。

### 3.4 代码实现要点

修改文件建议：

```text
utils/parser.py
network/Buffer.py
trainer/trainer.py
run_rhl_norm.sh 或新增 run_boa_rhl.sh
```

`RandomBuffer` 需要保留原始默认行为：

```text
rhl_init=gaussian
rhl_scale_mode=legacy
```

新增初始化函数：

```python
def _init_block_orthogonal(rows, cols, device, dtype):
    blocks = []
    remain = rows
    while remain > 0:
        q, _ = torch.linalg.qr(torch.randn(cols, cols, device=device, dtype=dtype))
        blocks.append(q[:min(cols, remain)])
        remain -= blocks[-1].shape[0]
    return torch.cat(blocks, dim=0)
```

antithetic 逻辑：

```python
half = math.ceil(out_features / 2)
base = _init_block_orthogonal(half, in_features, device, dtype)
weight = torch.cat([base, -base], dim=0)[:out_features]
```

必须保存这些参数进 checkpoint 的 `opts` 或日志文件，避免后续分不清模型来源。

### 3.5 完整工作流和优先级

| 步骤 | 动作 | 通过标准 |
|---|---|---|
| BOA-A | 实现参数和初始化，默认不改变 baseline | baseline 指标不变 |
| BOA-B | 跑 4 个单模型实验 | 至少一个 all mIoU 超 baseline `+0.10` 或 new mIoU 超 `+0.30` |
| BOA-C | 若 BOA-B 有效，补 seed2/3 | seed 方差下降或均值提升 |
| BOA-D | 若 BOA-C 有效，再做 RHL-SE | ensemble 超 best single `+0.05` |

优先级：P1，建议作为 RHL-SE 之后的第一主线。

### 3.6 预期结果

保守预期：

```text
all mIoU: -0.2 到 +0.4
new mIoU: -0.1 到 +0.8
seed 方差: 有机会下降
```

判定规则：

```text
如果 BOA 单模型没有超过 baseline，且 feature 诊断也没有改善，则停止；
如果 BOA 提升 new 但轻微损 old，考虑 old/new group-wise 与 baseline 融合；
如果 BOA 单模型有效，再考虑写成主要方法。
```

## 4. 方案三：PGH-RHL-lite

### 4.1 原始动机

原始 RHL 是纯随机特征：

```text
E_random = ReLU(F W^T)
```

它不知道类别结构。PGH-RHL 的动机是把少量类别原型相似度加入 RHL 表示：

```text
E_hybrid = [E_random, lambda * G_proto]
```

其中 `G_proto` 是像素特征到类别原型的相似度。

### 4.2 攻击审查

#### 问题 1：原型收集容易引入标签协议风险

VOC 15-5 sequential 下标签相对完整，但持续学习任务中不同 setting 的标签协议并不总是干净。PGH-RHL 必须先限定：

```text
第一版只在 VOC 15-5 sequential 做
只用训练集标签收集原型
不使用 val/test 标签或预测结果调原型
```

#### 问题 2：背景和边界像素会污染原型

语义分割的像素类别不均衡很严重。背景、边界、遮挡区域会把类别均值拉偏。

如果直接对所有像素求均值，原型可能不是“类别中心”，而是“数据集中常见背景上下文”。

#### 问题 3：RBF sigma 比是否加原型更关键

如果 sigma 太小，大多数像素相似度接近 0；如果 sigma 太大，所有类别相似度接近常数。两者都会让原型特征失效。

因此 PGH-RHL 不能只实现 `centers`，必须有 sigma 估计和诊断。

#### 问题 4：增加 21 维原型特征可能被 8196 维随机特征淹没

如果直接拼接：

```text
8196 random dims + 21 prototype dims
```

闭式解可能主要依赖随机维度。需要 `proto_scale` 控制原型特征权重。

### 4.3 重构后的方案

第一版只做 PGH-RHL-lite，不做多原型、不做 k-means。

#### 原型收集

数据来源：

```text
step0 train set 用于旧类原型
step1 train set 用于新类原型
```

限制：

```text
只用 train split
每类最多采样固定数量像素，例如 20000
忽略 ignore label
第一版不收集背景类或单独记录背景但默认不使用
```

如果必须覆盖背景：

```text
background prototype 只作为诊断，不参与第一版 hybrid feature
```

#### 原型特征

推荐第一版使用 cosine similarity，而不是 RBF：

```text
g_c(f) = cosine(normalize(f), normalize(mu_c))
```

原因：

```text
不需要先猜 sigma
尺度更稳定
与此前 RHL 归一化失败经验兼容
实现和诊断更简单
```

然后映射到非负：

```text
g_c^+(f) = (g_c(f) + 1) / 2
```

最终：

```text
E_hybrid = concat(ReLU(F W^T), proto_scale * G_cos)
```

如果 cosine 版有效，再考虑 RBF 版。

#### proto_scale

第一轮只测：

```text
proto_scale = 1.0
proto_scale = sqrt(buffer_size / num_proto)
```

第二个设置的作用是避免 21 维原型在数值上被 8196 维随机特征淹没。

### 4.4 代码实现要点

修改文件建议：

```text
network/Buffer.py
trainer/trainer.py
utils/parser.py
```

新增参数：

```text
--rhl_proto_mode none
--rhl_proto_mode cosine
--rhl_proto_scale 1.0
--rhl_proto_max_pixels 20000
--rhl_proto_include_bg false
```

新增模块：

```python
class PrototypeCosineBuffer(nn.Module):
    def __init__(self, centers, scale):
        self.register_buffer("centers", F.normalize(centers, dim=1))
        self.scale = scale

    @torch.no_grad()
    def forward(self, x):
        x = F.normalize(x, dim=-1)
        sim = torch.matmul(x, self.centers.t())
        return (sim + 1.0) * 0.5 * self.scale
```

混合 buffer：

```python
class HybridRandomPrototypeBuffer(nn.Module):
    def forward(self, x):
        return torch.cat([self.random_buffer(x), self.prototype_buffer(x)], dim=-1)
```

`AIR` 的 analytic input dim 必须跟随 buffer 输出维度：

```text
analytic_in_features = random_buffer_size + num_prototypes
```

保存 checkpoint 时必须包含：

```text
prototype centers
prototype class ids
proto_scale
proto_mode
```

### 4.5 完整工作流和优先级

| 步骤 | 动作 | 通过标准 |
|---|---|---|
| PGH-A | 只实现 prototype collection 并保存 centers | 每个新类 count 正常，center 非 NaN |
| PGH-B | 加 cosine prototype buffer，跑 `max_batches` smoke | 输出维度正确，checkpoint 可加载 |
| PGH-C | 跑 `proto_scale=1.0` 单模型 | 不明显低于 baseline |
| PGH-D | 跑 `proto_scale=sqrt(8196/20)` 单模型 | new mIoU 有可见提升 |
| PGH-E | 若有效，补 old/new 融合或 BOA+PGH | all mIoU 超 baseline |

优先级：P2。PGH 比 BOA 更有论文方法感，但工程和协议风险更高。

### 4.6 预期结果

保守预期：

```text
all mIoU: -0.3 到 +0.6
new mIoU: +0.0 到 +1.0
old mIoU: 可能轻微下降
```

判定规则：

```text
如果 new 涨但 old 掉，考虑只对新类 logits 使用 PGH 分支；
如果 all/new 都不涨，说明 frozen feature 的类别原型没有提供额外信息；
如果 val 有效 test 无效，优先检查 proto_scale 过拟合和类别像素采样偏差。
```

## 5. 三方案整合后的推荐路线

### 5.1 总路线

```text
阶段 0A：RHL-SE 收尾
  val-driven class-wise 权重搜索
  若收益仍小于 +0.10 all mIoU，则定位为辅助模块

阶段 0B：BOA-RHL single
  边界清晰的代码改动
  先证明随机特征构造本身能提升

阶段 1：PGH-RHL-lite
  当 BOA 或 RHL-SE 的证据显示仍需要更强类别结构时，加入类别原型语义锚点

  只作为结项系统级集成和上界补充
```

执行原则：

```text
P0 与 P1 同轮推进。
```

## 6. 预期论文表述方式

如果 BOA-RHL 有效：

```text
我们发现原始 RHL 的随机基函数存在方向冗余和 ReLU 单侧截断问题。
因此提出 block-orthogonal antithetic RHL，在保持闭式解增量学习框架不变的情况下，提高固定随机特征映射的覆盖质量。
```

如果 PGH-RHL 有效：

```text
我们进一步指出纯随机 RHL 缺少类别结构先验。
因此提出 prototype-guided hybrid RHL，将训练集统计得到的类别原型相似度作为少量语义锚点特征，与随机高维特征拼接后共同进入 C-RLS 闭式解。
```

如果 Snapshot 只作为系统补充：

```text
为满足集成系统需求，我们还构建了基于不同 step0 feature extractor snapshot 的 analytic ensemble。
该模块不改变主方法，而作为系统级增强和上界分析。
```

## 7. 最关键的止损规则

为避免再次陷入“理论能解释但指标不动”的循环，后续每个方案都必须有止损规则。

| 方案 | 继续条件 | 停止条件 |
|---|---|---|
| RHL-SE | val-driven test 提升达到 `+0.10 all` 或 `+0.30 new` | 低于阈值则只作为辅助模块 |
| BOA-RHL | 单模型超过 baseline `+0.10 all` 或 `+0.30 new` | 4 组实验都无提升则停止 |
| PGH-RHL | new 明显提升且 old 不大跌 | val 有效 test 无效或 old 大跌则重查协议 |
| Snapshot | ensemble 超 best single `+0.10 all` | 成员高度相关或质量低则停止 |

## 8. 最终建议

下一轮最值得执行的是：

```text
BOA-RHL single
```

原因：

1. 它仍然是 RHL 机制内改造，不会像 Snapshot 那样偏到系统集成。
2. 它直接改变 RHL 特征构造，比归一化和推理权重更接近误差源。
3. 工程量低于 PGH-RHL，可以快速验证。
4. 若单模型有效，可自然与 RHL-SE 或 old/new group-wise 融合结合。

PGH-RHL-lite 是更有论文味道的第二候选，但应放在 BOA 之后。Snapshot 应保留为结项和补充实验，不建议作为主要方法突破口。
