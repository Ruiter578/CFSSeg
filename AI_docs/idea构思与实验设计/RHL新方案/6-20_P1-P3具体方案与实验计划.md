# P1-P3 具体方案与实验计划：BOA-RHL 1.5、PGH-RHL-lite 与 Snapshot Analytic Ensemble

> 日期：2026-06-20
> 状态：P1 第一轮已完成；P1.5、P2、P3 待实现与实验
> 统一任务：VOC 15-5 sequential step1，除非实验表明确改变单一变量

## 0. 范围与术语

本文重新核对 P1、P2、P3 的方法动机、代码边界和实验协议，并保留各方法的完整研究空间，不因第一轮失败或优先级不同而删除 antithetic、多原型、k-means、RBF sigma 搜索或 Snapshot。

统一术语为 **PGH-RHL-lite（Prototype-Guided Hybrid RHL）**。任务描述中的“RGH-RHL-Lite”在现有设计文档和代码语境中对应 PGH-RHL-lite，本文将其视为同一方法的别名/笔误，不再建立第二个含义不明的方法名。

三条线的边界：

- P1 BOA-RHL：改随机映射基的几何结构；
- P2 PGH-RHL-lite：在随机特征之外增加有监督语义原型响应；
- P3 Snapshot：改变上游 backbone checkpoint，制造结构级成员多样性。

## 1. P1：BOA-RHL 1.5

### 1.1 第一轮事实

| Case | init | scale | all | old | new | 判断 |
|---|---|---|---:|---:|---:|---|
| BOA-0 | `gaussian` | `legacy` | 69.5598 | - | 43.2099 | 精确复现强基线 |
| BOA-1 | `orthogonal` | `legacy` | 69.6387 | 77.8225 | 43.4507 | 唯一正向信号 |
| BOA-2 | `orthogonal_antithetic` | `legacy` | 68.3424 | - | 38.3768 | 明显退化 |
| BOA-3 | `orthogonal_antithetic` | `kaiming` | 68.1668 | - | 37.7973 | scale 未救回退化 |

BOA-0 复现说明新增分支未破坏默认路径。BOA-1 相对 BOA-0 的 all/new 约 `+0.0789/+0.2408`，信号为正但不足以排除随机波动。BOA-2/3 无 NaN、Inf、OOM，失败来自表示能力或有效维度，不是程序崩溃。

现有 `gaussian` 名称应被视为历史兼容名：legacy 路径实际调用 `torch.nn.Linear.reset_parameters()`，是 Kaiming-uniform 风格，并非真正 Gaussian。后续报告必须写成“legacy baseline (`gaussian` CLI alias)”，避免错误的理论比较。

### 1.2 P1.5 的核心问题

1. `orthogonal/legacy` 的收益能否跨 RHL seed 重复？
2. 收益来自方向去相关，还是仅来自行范数/激活尺度变化？
3. antithetic 的退化是否主要由独立方向数减半造成？增加 BUFFER 能否恢复？
4. 更大 BUFFER 带来的提升，是否只是容量提升而非 antithetic 机制提升？

### 1.3 P1.5-A：随机种子复核

固定以下所有因素：global seed、base checkpoint、Batch Size、Buffer、gamma、norm、训练轮次，只改变 `rhl_seed=1,2,3`。

| Case | init | scale | buffer | RHL seeds |
|---|---|---|---:|---|
| A0 | legacy baseline | legacy | 8196 | 1,2,3 |
| A1 | orthogonal | legacy | 8196 | 1,2,3 |

报告每个 seed 的 all/old/new/per-class、mean/std，以及相对同 seed baseline 的 paired delta。主判断使用 paired delta，不用“最好的单次结果”。

### 1.4 P1.5-B：尺度解耦与 gamma 匹配

随机特征整体乘以 `s` 时，Ridge 中的有效正则强度会随 `s^2` 改变。若 baseline 特征为 `H`，新特征为 `sH`，为了比较方向结构而非正则变化，使用：

```text
gamma_new = gamma_base * s^2
```

按当前行范数定义，近似关系为：

- legacy 行范数约 `1/sqrt(3)`；
- unit 行范数为 `1`，相对 scale `sqrt(3)`，匹配 gamma 约 `3`；
- kaiming 行范数约 `sqrt(2)`，相对 scale `sqrt(6)`，匹配 gamma 约 `6`。

实验矩阵：

| Case | init | scale | gamma | 目的 |
|---|---|---|---:|---|
| B0 | orthogonal | legacy | 1 | 已有参照 |
| B1 | orthogonal | unit | 1 | 观察方向+尺度共同变化 |
| B2 | orthogonal | unit | 3 | 近似匹配有效正则 |
| B3 | orthogonal | kaiming | 1 | 观察方向+尺度共同变化 |
| B4 | orthogonal | kaiming | 6 | 近似匹配有效正则 |

同时记录随机特征 `mean/std/row_norm/input-output trace`，不要只记录最终 mIoU。

### 1.5 P1.5-C：BUFFER 与 antithetic 独立方向审计

`orthogonal_antithetic` 通过 `[W; -W]` 构造特征。总输出 8196 时，只有 4098 个独立方向。必须保留并验证“大 BUFFER 恢复独立方向”的假设，但实验要区分两种公平性：

1. **同总维度公平**：orthogonal 16392 vs antithetic 16392；
2. **同独立方向公平**：orthogonal 8196 vs antithetic 16392。

建议矩阵：

| Case | init | buffer | 独立方向 | 目的 |
|---|---:|---:|---:|---|
| C0 | baseline | 8196 | 8196 | 容量基线 |
| C1 | orthogonal | 8196 | 8196 | BOA-1 复核 |
| C2 | baseline | 12288 | 12288 | 容量趋势 |
| C3 | orthogonal | 12288 | 12288 | 中等容量 BOA |
| C4 | baseline | 16392 | 16392 | 大容量控制 |
| C5 | orthogonal | 16392 | 16392 | 同总维度控制 |
| C6 | antithetic | 16392 | 8196 | 恢复独立方向 |

16392 不是普通训练直接开跑项。`RecursiveLinear` 当前使用 double 精度，`R` 的单个方阵约占 `16392^2*8` bytes，约 2.15 GB，逆矩阵和临时算子显存会显著更大。执行顺序必须是：

1. 初始化与单 batch forward；
2. 单 batch `fit` 并记录峰值显存、矩阵条件代理和时延；
3. 通过后再完整训练；
4. 若不可承受，先实现分块/低秩求解或使用 12288，不得把 OOM 解释为方法无效。

### 1.6 BOA 诊断增强

新增并落盘：

- 随机矩阵行范数分布；
- block 内/跨 block coherence；
- 特征协方差 effective rank；
- `H^T H` 的 trace、最大/最小特征值近似、condition proxy；
- 各类别特征中心间距；
- seed 间 paired delta。

可使用随机投影、幂迭代或抽样估计，避免为完整 SVD 引入不可承受开销。

## 2. P2：PGH-RHL-lite

### 2.1 动机

纯随机特征提供非线性展开，但没有任何“该像素接近哪一类”的显式语义锚点。PGH-RHL-lite 在不放弃解析学习主体的前提下，引入从训练集提取的类别原型响应：

```text
backbone feature F
  -> random hidden feature H_rand
  -> prototype response H_proto
  -> branch calibration
  -> concat([H_rand, H_proto])
  -> RecursiveLinear
```

这里的原型不是新的可训练分类器，而是固定语义特征映射。核心研究问题是：少量、可解释的 prototype response 能否弥补纯随机映射对新类结构表达不足。

### 2.2 标签与数据协议

原型只允许从训练数据构建：

- old-class prototypes：step0 train 数据；
- new-class prototypes：step1 train 数据；
- 禁止使用 val/test image 或 label；
- 以原始训练 split 为数据源，但使用专用确定性 prototype transform，不能使用随机训练 crop/flip；
- `shuffle=False`、`drop_last=False`，固定抽样 seed；
- 默认排除 ignore label 和边界不确定像素；
- background 第一阶段不入库，后续作为独立消融。

原型构建器必须读取 VOC sequential label mapping 的真实语义，并在缓存 metadata 中记录每类像素数量、来源 step 和 transform。不能假定 step1 mask 只含新类。

### 2.3 原型库构建

对每个类别 `c` 收集 L2-normalized backbone pixels `f_i`。单原型：

```text
p_c = normalize(mean_i(normalize(f_i)))
```

多原型使用离线 `MiniBatchKMeans`：

```text
{p_c1, ..., p_cK} = KMeans({normalize(f_i)}, K)
```

每类使用 reservoir sampling 或固定上限，避免大类吞噬内存。建议参数：

```text
--rhl_proto_per_class 1|2|4
--rhl_proto_max_pixels 50000
--rhl_proto_kmeans_seed 1
--rhl_proto_include_bg false|true
--rhl_proto_boundary_erode 0|1|2
```

第一轮 `K=1`，第二轮 `K=2/4`。多原型与 k-means 是完整方案的一部分，不删除，只放在缓存正确性和单原型信号验证之后执行。

### 2.4 原型响应

#### Cosine 分支

```text
h_cj(x) = cosine(normalize(F(x)), p_cj)
```

优点是无需 sigma，适合验证原型信息是否有效。

#### RBF 分支

```text
h_cj(x) = exp(-||normalize(F(x)) - p_cj||^2 / (2*sigma_cj^2))
```

sigma 模式完整保留：

- `global_median`：所有训练样本距离的全局中位数；
- `class_median`：每类独立中位数；
- `prototype_local`：每个原型局部距离统计；
- multiplier 大搜索：`0.25, 0.5, 1, 2, 4`，只用 validation 选择。

RBF 大搜索在 cosine 与 K=1 流程验证后进行，避免把缓存错误、分支尺度错误和 sigma 错误混在一起。

### 2.5 分支尺度与容量公平

21 或 84 维 prototype response 与 8196 维 random feature 直接拼接时可能被淹没。定义：

```text
H = concat(H_rand, alpha_proto * H_proto)
```

`alpha_proto` 不只做任意网格搜索，先用 branch RMS/trace 匹配初始化：

```text
alpha_proto = sqrt(trace(H_rand^T H_rand) / trace(H_proto^T H_proto))
              * target_ratio
```

实验同时覆盖：

- `budget_mode=additive`：8196 随机维 + prototype 维；
- `budget_mode=fixed`：总维度固定 8196，从随机维中扣除 prototype 维。

前者测“增加语义分支是否有用”，后者测“同容量下语义维是否优于随机维”。

```text

--rhl_proto_bank PATH
--rhl_proto_per_class INT
```

`none` 必须复现原 RHL。原型库 `.pt` 与 metadata JSON 必须包含 base checkpoint/hash、dataset/task/step、类别列表、transform、每类计数、K、seed、sigma、dtype、shape 和 checksum。

### 2.7 PGH 实验阶段

#### P2-A：缓存与单原型 cosine

| Case | K | proto | budget | scale |
|---|---:|---|---|---|
| P2-A0 | 0 | none | 8196 random | - |
| P2-A1 | 1 | cosine | additive | trace_match |
| P2-A2 | 1 | cosine | fixed | trace_match |
| P2-A3 | 1 | cosine | additive | 0.5x/1x/2x trace |

#### P2-B：多原型与 k-means

| Case | K | proto | budget | 目的 |
|---|---:|---|---|---|
| P2-B1 | 2 | cosine | additive/fixed | 类内双模态 |
| P2-B2 | 4 | cosine | additive/fixed | 更细粒度类内结构 |

检查 prototype utilization：每个原型被选中的像素比例、类内最近原型距离，以及是否出现空簇/重复簇。

#### P2-C：RBF sigma 完整搜索

对 K=1/2 的最佳配置执行三种 sigma 模式和 multiplier `0.25/0.5/1/2/4`。先在 val 选择每种模式，再对冻结的最佳 1-2 个配置执行 test。K=4 的 RBF 只在 K=4 cosine 展示额外价值时执行，避免无信息的组合爆炸，但该分支保留为完整后续实验。

#### P2-D：与 BOA 的组合

仅当 BOA 与 PGH 均在独立实验中有稳定正信号后，做 2x2 因子实验：

| | PGH off | PGH on |
|---|---|---|
| BOA off | baseline | PGH |
| BOA on | BOA | BOA+PGH |

交互项：

```text
interaction = gain(BOA+PGH) - gain(BOA) - gain(PGH)
```

这能区分简单叠加、互补增强与相互干扰。

## 3. P3：Snapshot Analytic Ensemble

### 3.1 方法定位

Snapshot 不是 RHL 随机特征本身的改进，而是上游 backbone 多样性方案。它利用 step0 训练中不同 epoch 的模型形成成员，再让每个成员走相同 step1 解析训练和最终集成。

当前目录中大量 `step0/final.pth` 是对齐后的 AIR 产物或最终 checkpoint，不等于训练过程 Snapshot。现有训练也没有系统保存中间 epoch，因此 P3 不能直接宣称已有可用 Snapshot。

### 3.2 checkpoint 基础设施

新增：

```text
--snapshot_epochs 20,30,40,50
--snapshot_interval INT
--base_ckpt PATH
--snapshot_manifest PATH
```

`--base_ckpt` 的加载优先级高于 `BASE_SUBPATH`，目的是让每个 step1 run 明确引用只读 Snapshot，而不是复制并覆盖某个 `final.pth`。

Snapshot manifest 记录：

- checkpoint 路径/hash；
- epoch/iteration；
- step0 val all/old/per-class；
- model config；
- optimizer/scheduler state 是否存在；
- git commit 与训练 run id。

### 3.3 Snapshot 选择

不采用盲目 last-k。候选集合先满足基本质量下限，再根据“质量 + 多样性”选 3 个：

```text
score(S) = mean_val_mIoU(S)
           + lambda_div * pairwise_prediction_diversity(S)
           - lambda_bad * weak_member_penalty(S)
```

多样性可在固定 validation probe set 上计算：

- prediction disagreement；
- logit cosine distance；
- per-class IoU 向量距离；
- backbone feature CKA/cosine 的近似统计。

最终选择不应包含明显弱于最佳 Snapshot 的早期模型，除非其加入后 validation ensemble 明确更好。

### 3.4 Step1 与集成协议

第一轮所有 Snapshot 使用完全相同的 step1 配置：

- 同一 RHL seed；
- 同一 init/scale/norm/buffer/gamma；
- 同一 Batch Size 与数据顺序；
- 唯一变化是 `base_ckpt`。

先比较每个成员的单模型结果，再在 val 选择：uniform、global weight、RHL-SE 2.0。冻结成员和权重后执行一次 test。

### 3.5 P3 实验矩阵

| 阶段 | 实验 | 目的 |
|---|---|---|
| P3-A | 保存并审计 step0 snapshots | 建立真实候选集 |
| P3-B | 质量 top3 vs 多样性 top3 | 验证选择准则 |
| P3-C | 每个 Snapshot + plain RHL | 隔离上游多样性收益 |
| P3-D | uniform/class-wise/reliability gate | 比较融合策略 |
| P3-E | Snapshot + 最佳 BOA 或 PGH | 系统级最终组合 |

## 4. P1-P3 的依赖与并行关系

```text
P1 BOA ---------+
                +--> BOA+PGH 单成员 --> RHL-SE 2.0
P2 PGH ---------+

P3 Snapshot --> 多个 backbone --> plain/BOA/PGH step1 --> RHL-SE 2.0
```

- P1 与 P2 可并行设计和实现；它们都修改 hidden feature 构造，组合时必须统一维度和 branch scale。
- P3 不依赖 P1/P2 才能启动，第一轮应使用 plain RHL 隔离 Snapshot 效果。
- P3 后续可把每个 Snapshot 的 step1 方法替换为 P1/P2 最佳配置。
- RHL-SE 2.0 是下游融合器，不作为 P1/P2/P3 单方法有效性的前置条件。

## 5. 推荐执行工作流

1. 完成 BOA P1.5-A seed 复核，同时实现 BOA 诊断统计。
2. 执行 P1.5-B scale/gamma 匹配；根据资源 smoke test 推进 P1.5-C BUFFER 审计。
3. 与 P1 并行建立 PGH prototype bank 和缓存测试，完成 P2-A cosine。
4. 推进 P2-B 多原型/k-means，再做 P2-C RBF sigma 完整搜索。
5. 用独立正向配置做 BOA+PGH 2x2 因子实验。
6. 增加 Snapshot 保存/显式 base checkpoint 基础设施，训练候选 step0 snapshots。
7. 完成 Snapshot 质量/多样性选择、plain RHL step1 和融合实验。
8. 最后把最佳单成员方法、Snapshot 成员和 RHL-SE 2.0 组合，形成系统结果。

优先级决定先后和资源分配，不代表后续阶段被删除。每条方法线都必须保留独立 baseline、开关、配置持久化和单方法结果，组合实验只在这些证据完整后解释交互。
