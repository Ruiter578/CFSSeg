## 4. 面向小白：buffer 到底是什么

### 4.1 一句话解释

这里的 `buffer` 不是“样本缓存池”，不是 replay memory，也不是保存旧图片的地方。它是 AIR/RHL 模块里的“随机高维特征维度”。

更直白地说：

> 模型先用 DeepLab 提取每个像素的 256 维特征，然后用一个固定随机矩阵把 256 维特征投影到 `buffer` 维。`buffer=8192` 就是每个像素被扩展成 8192 维随机特征；`buffer=8224` 就是扩展成 8224 维。

### 4.2 它在训练流程的哪个阶段发挥作用

在 step1 里，流程大致是：

```text
输入图像
  -> DeepLab backbone / decoder 提取 dense feature
  -> 每个像素得到 256 维特征
  -> RandomBuffer / RHL 固定随机投影
  -> 每个像素变成 buffer 维特征
  -> RecursiveLinear 用闭式/递推方式求分类器权重
  -> 输出每个像素的类别分数
```

它主要在 `curr_step=1` 的 AIR 替换阶段发挥作用。代码路径是：

- `trainer/trainer.py:397-410` 构造 `AIR(...)`
- `trainer/trainer.py:47-55` 里面创建 `RandomBuffer(...)` 和 `RecursiveLinear(...)`
- `trainer/trainer.py:66-72` 先取 backbone 特征，再过 `self.buffer(X)`
- `trainer/trainer.py:94-98` 在 `fit()` 中把随机高维特征交给解析分类器

### 4.3 它在模型架构的什么位置

它夹在 DeepLab 特征提取器和解析线性分类器之间：

```text
DeepLab feature extractor
  输出: [B, 256, Hf, Wf]

reshape
  输出: [B, Hf*Wf, 256]

RandomBuffer / RHL
  输出: [B, Hf*Wf, buffer_size]

RecursiveLinear
  输出: [B, Hf, Wf, num_classes]
```

所以 `buffer_size` 不是类别数，也不是 batch size。它控制中间随机特征空间的宽度。

### 4.4 为什么 buffer 维度会影响结果

可以把 `RandomBuffer` 想成很多个固定的随机探测器。每个探测器看原始 256 维特征的一个随机方向，然后经过 ReLU 非线性。`buffer` 越大，随机探测器越多，特征表达能力通常越强。

但它不是越大越好，原因有三个：

1. 随机投影有运气成分。同样的维度，不同 `rhl_seed` 会生成不同随机矩阵。
2. `RecursiveLinear` 要处理 `buffer_size x buffer_size` 级别的矩阵，维度越大，数值条件、显存、计算成本都更敏感。
3. `gamma` 是岭回归正则，`buffer` 改变后，特征能量和正则强度的相对关系也可能改变。

### 4.5 gamma 是什么

`gamma` 是解析线性分类器里的正则项。`RecursiveLinear` 初始化：

```python
R = torch.eye(self.weight.shape[0]) / self.gamma
```

后续更新等价于带正则的最小二乘：

```text
min_W ||Y - XW||^2 + gamma * ||W||^2
```

直觉上：

- `gamma` 太小：分类器更容易贴合训练数据，但可能数值不稳定或过拟合。
- `gamma` 太大：分类器更保守，可能压低新类学习。
- 本轮按你的要求固定 `gamma=1`，所以当前结论是“在 gamma=1 条件下”的 buffer 结论。

### 4.6 rhl1 / rhl2 / rhl3 是什么

`rhl1/rhl2/rhl3` 不是训练第 1/2/3 阶段，也不是 step 编号。它们是 `rhl_seed=1/2/3` 的文件名简写。

含义是：

```text
rhl_seed=1 -> 用 1 作为 RandomBuffer 随机矩阵的种子
rhl_seed=2 -> 用 2 作为 RandomBuffer 随机矩阵的种子
rhl_seed=3 -> 用 3 作为 RandomBuffer 随机矩阵的种子
```

代码里 `RandomBuffer` 对 `rhl_seed>=0` 使用 `torch.random.fork_rng`，只临时改变 RHL 初始化种子，退出后恢复外部 RNG。这样它理论上只改变随机投影矩阵，不改变 DataLoader 顺序、数据增强和其他随机过程。

文件名里的 `rhlm1` 表示 `rhl_seed=-1`，`m1` 是 minus 1 的意思。

### 4.7 为什么要做多个 rhl_seed

因为 RHL 是随机映射，单次结果可能有“抽到一个好随机矩阵”的运气。多 seed 是为了回答：

> 这个 buffer 真的稳定更好，还是刚好某个随机投影矩阵比较幸运？

这就是 Phase B 的意义。当前 Phase B 显示 `8224` 在 `rhl_seed=2/3` 上明显强于 `8208`，但它的方差也更大。

## 5. 8192、8208、8224 应该怎么理解

论文里的 `8192` 更像一个稳健、工程上常见的默认值。它是 2 的幂，显存和矩阵形状都规整，也便于复现。

本地实验显示：

- `8192` 在当前协议下不是最优。
- `8208` 在单 seed Phase A 里最好。
- `8224` 在多 RHL seed 后更好。

但这不等于“论文默认值应该改成 8224”。原因是论文默认值需要跨任务、跨数据集、跨随机种子、跨实现设置稳定成立。当前证据只能支持一个更窄的结论：

> 对本仓库当前 `main@99c0b5f + dirty working tree`、VOC `15-5 sequential` step1、`20260621_baseline_bs16_16_trs` step0 权重、`gamma=1`、`rhl_norm=none` 这组条件，`8224` 是当前最值得继续验证的 buffer 候选。

## 6. 深度讨论：为什么 8224 看起来更值得继续

### 6.1 old 类基本不动，new 类驱动提升

`8208` 和 `8224` 的 old 类均值都在 `0.784` 左右，差距很小：

| buffer | mean old 0-15 mIoU |
|---:|---:|
| 8208 | 0.784566 |
| 8224 | 0.784277 |

但 new 类均值差距更明显：

| buffer | mean new 16-20 mIoU |
|---:|---:|
| 8208 | 0.421462 |
| 8224 | 0.425188 |

所以 `8224` 的优势主要来自新类学习，而不是旧类保持。这对增量分割很关键，因为 step1 的痛点通常就是新类和旧类的平衡。

### 6.2 8224 更强但更不稳定

`8224` 的 mean all 更高，但 std 也更高：

| buffer | mean all | std all |
|---:|---:|---:|
| 8208 | 0.698112 | 0.000691 |
| 8224 | 0.698780 | 0.001403 |

这说明 `8224` 更像“上限更高，但随机投影质量更敏感”。如果后续 seed 继续支持它，就能把结论变强；如果 seed 4/5 回落，则应更保守地把 `8208` 当成稳定候选。

### 6.3 当前搜索可能还没到右边界

Phase A 到 `8224` 截止，但 Phase B 后 `8224` 成了最强候选。这种情况在实验设计上叫“最优点贴边”。贴边时，下一步应该向边界外延伸，而不是直接宣布最优。

### 7.2 第二优先级：补足 8208 与 8224 的 seed 4/5

理由：当前只有 3 个唯一 seed，且 `8224` 方差较大。补 seed 4/5 可以判断它是否稳定。

| buffer | gamma | rhl_seed |

这组比向右扩展更保守，适合目标是“确认 8208 vs 8224 谁更稳”。如果目标是“找更高上限”，优先跑 7.1。

### 7.3 暂不建议启动的实验

暂不建议继续跑 `gamma=0.1/10`。理由不是它没价值，而是你已经明确要求后续固定 `gamma=1`。当前最应该解决的问题是：

1. `8224` 是不是边界假象。
2. `8224` 的 seed 方差是否可接受。
3. 更大的 buffer 是否会继续提升新类 mIoU。

## 8. 推荐下一步决策

如果现在只允许启动一组实验，我建议启动“右边界扩展”：

```text
buffer = 8228, 8232, 8236, 8240
gamma = 1
rhl_seed = 2
其他变量完全沿用本轮设置
```

理由：

- 当前最强单次是 `8224/rhl2`，所以用 `rhl_seed=2` 可以做最直接的右侧对比。
- 如果右侧继续提升，说明之前的搜索范围太窄。
- 如果右侧不提升，再回头补 `8208/8224` 的 seed 4/5，结论会更干净。

最终论文式表述暂时不要写成“buffer=8224 是最优”。建议写成：

> 当前实验发现，将 RHL buffer 从论文默认 8192 扩大到 8208-8224 区间可提升 VOC 15-5 step1 的 mIoU；在固定 `gamma=1` 下，`8224` 在多 RHL seed 检查中取得最高均值，但由于该点位于搜索上边界，仍需进一步向右扩展验证。
