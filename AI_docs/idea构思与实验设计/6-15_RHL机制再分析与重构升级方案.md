## 0. 总结结论

RHL 是 CFSSeg 的关键突破口，但此前做的 **RHL 输出归一化** 不是足够强的突破口。

原因是：原论文中 RHL 的贡献主要来自 **随机非线性高维基函数带来的线性可分性提升**，而归一化只改变高维特征的尺度，不增加新的基函数、不引入类别结构、不修正标签噪声，也可能抹掉 RHL 特征范数中有用的置信度和 objectness 信息。

后续更建议围绕 RHL 的三个核心属性做改造：

| 优先级 | 方案 | 核心思想 | 推荐程度 |
|---:|---|---|---:|
| P0 | **RHL Subspace Ensemble, RHL-SE** | 多个 RHL 随机子空间分别闭式求解，推理时概率平均 | 高 |
| P1 | **Block-Orthogonal Antithetic RHL, BOA-RHL** | 用正交块和反向成对随机特征降低随机特征方差 | 中高 |
| P2 | **Prototype-Guided Hybrid RHL, PGH-RHL** | 在随机 RHL 外加入类别原型 RBF 特征，使 RHL 从纯随机变成随机加语义锚点 | 中 |

如果只能先做一个，建议先做 **RHL-SE**。它最符合项目“集成学习与集成系统”的结项口径，工程成本低，且和 RHL 机制天然一致。

### 0.1 范围边界补充（2026-06-16 修订）

本文件有意把创新范围收束在 RHL 上，不是因为全局方案不重要，而是因为当前任务要求是“围绕 RHL 这一关键机制重新构思”。从全局角度看，确实还存在更宽的 ensemble 选择，例如：

```text
全局 random_seed ensemble
step0 backbone snapshot ensemble
DeepLabV3+ / backbone 对照
TTA 推理集成
```

这些方案可能在实测上带来更大的多样性，尤其是直接改变全局 `random_seed` 时，会同时改变数据增强、DataLoader 顺序、RHL 初始化和若干其他随机过程，ensemble 上限可能高于“只改 RHL seed”。但是它们的因果解释不再干净：如果结果上涨，无法说明收益来自 RHL 子空间。

因此本文的定位应理解为：

```text
RHL-SE 是 RHL 机制内的干净可解释集成；
全局 seed ensemble 是更宽范围的系统集成对照或后续增强版；
二者不是互斥关系，但论文中应分开命名、分开实验、分开解释。
```

后续执行时，第一优先级仍建议固定全局 `random_seed=1`，只改变独立的 `rhl_seed`。若 RHL-SE 效果有限，再补充 `global-seed ensemble` 作为“系统级集成上界”对照，而不是把它混入 RHL-SE 的主结论。

## 1. 原论文中 RHL 的真实动机

CFSSeg 的核心链路是：

```text
step0: 用 SGD/BP 训练 DeepLabV3 + ResNet101，得到基础 encoder
step>0: 冻结 encoder，不再用梯度更新 backbone
        dense feature -> RHL -> C-RLS 闭式分类头
```

原论文对 RHL 的动机很明确：

1. 冻结 encoder 能带来稳定性，避免新任务梯度破坏旧知识。
2. 冻结 encoder 会牺牲可塑性，新类可能难以被原始 256 维 dense feature 线性分开。
3. 根据 Cover 定理的直觉，非线性高维映射能提高样本线性可分概率。
4. 因此作者把 encoder feature 送入随机初始化隐藏层 RHL，再用 ReLU 得到高维特征。

数学形式是：

$$
E = \operatorname{ReLU}(F\Phi_E)
$$

其中：

| 符号 | 含义 |
|---|---|
| $F$ | 冻结 encoder 输出的 dense pixel feature |
| $\Phi_E$ | 随机初始化并固定的 RHL 映射矩阵 |
| $E$ | 进入 C-RLS 的高维随机特征 |

然后解析头做 ridge regression：

$$
\hat{\Phi} = (E^\top E + \gamma I)^{-1}E^\top Y
$$

在持续学习中，C-RLS 用递归形式保存历史二阶统计：

$$
\Psi_t = (\Psi_{t-1}^{-1} + E_t^\top E_t)^{-1}
$$

所以 RHL 在整个系统中的作用是：

```text
冻结 encoder 提供稳定性
RHL 提供可塑性和线性可分性
C-RLS 提供闭式更新和历史统计记忆
伪标签在 disjoint / overlapped 中修复 semantic drift
```

## 2. 为什么论文消融说明 RHL 很关键？

原论文在 Pascal VOC2012 overlapped 10-1 上做了 RHL 消融。结果非常强：

| 设置 | old mIoU | new mIoU | all mIoU |
|---|---:|---:|---:|
| Ours | 75.02 | 41.20 | 58.91 |
| w/o RHL | 63.91 | 9.36 | 37.94 |

这个结果说明：

```text
冻结 encoder 后，如果直接用原始 feature 做线性解析头，新类几乎学不动。
RHL 不是可选 trick，而是冻结表示之后恢复新类可塑性的必要模块。
```

但这也容易带来一个误解：

```text
RHL 很重要，所以任何对 RHL 的改动都会涨点。
```

这个推论不成立。论文消融证明的是：

```text
有 RHL >> 无 RHL
```

它没有证明：

```text
归一化 RHL > 原始 RHL
```

RHL 的核心收益来自“增加随机非线性高维基函数”。归一化没有增加新的基函数，只是改变这些基函数输出后的尺度。

## 3. 为什么 RHL 归一化没有带来性能提升？

当前已完成实验显示：

| 设置 | Batch | all mIoU | old mIoU | new mIoU |
|---|---:|---:|---:|---:|
| `none, gamma=1` | 64 | 69.461 | 78.008 | 42.107 |
| `l2, gamma=1` | 64 | 69.451 | 77.939 | 42.287 |
| `layernorm, gamma=1` | 64 | 69.415 | 77.957 | 42.079 |
| `l2_sqrt, gamma=1` | 64 | 69.304 | 77.943 | 41.657 |
| `l2_sqrt, gamma=0.1/1/10` | 64 | 69.304 | 77.943 | 41.657 |

从结果看，RHL 归一化确实生效，但没有带来可靠涨点。

### 3.1 归一化解决的是数值尺度，不是线性可分性

RHL 的目标是让类别在高维非线性空间里更容易线性分开。

归一化做的是：

$$
\tilde{e}_i = \frac{e_i}{\|e_i\|_2+\epsilon}
$$

或：

$$
\tilde{e}_i = \sqrt{d}\frac{e_i}{\|e_i\|_2+\epsilon}
$$

这能控制 row norm，但它不保证类间 margin 变大。

换句话说：

```text
RHL 的关键是“构造什么样的高维非线性特征”
归一化只是在“构造之后调尺度”
```

如果原始 RHL 已经能稳定求解，归一化并不会自动提升 mIoU。

### 3.2 RHL 特征范数可能不是噪声，而是有用信号

当前日志里，原始 `none` 的 row norm 并不恒定：

```text
mean 约 28-30
std 约 5-6
min 约 11-14
max 约 55-67
```

一开始我们把这种波动理解成数值不稳定来源。但在 dense segmentation 中，这些范数也可能包含有效信息：

| RHL 范数可能编码的信息 | 对分割的意义 |
|---|---|
| backbone 响应强度 | 目标像素 vs 背景像素 |
| 特征激活强弱 | 清晰区域 vs 边界区域 |
| 类别可分难度 | 易分样本 vs 难分样本 |
| 隐式置信度 | 高置信旧类或新类区域 |

`l2` 和 `l2_sqrt` 会把这些强弱信息抹平，使 C-RLS 只看到“方向”，看不到“响应强度”。

这会把 RHL 从一个 ReLU 随机特征映射变成更接近 cosine feature 的映射。对分类可能有时有利，但对像素分割不一定有利。

### 3.3 ReLU 随机特征的幅值本来就是 kernel 的一部分

RHL 可以理解成一种随机特征近似。对 ReLU 随机特征来说：

$$
k(f_i, f_j) \approx \operatorname{ReLU}(f_i W)\operatorname{ReLU}(f_j W)^\top
$$

这个相似度不仅依赖角度，也依赖 $f_i$ 和 $f_j$ 的范数。行归一化后，相似度更接近：

$$
\tilde{k}(f_i, f_j) =
\frac{e_i^\top e_j}{\|e_i\|_2\|e_j\|_2}
$$

也就是更偏 cosine similarity。

如果原始 ReLU random feature 的幅值正好反映了分割中的 objectness 或置信度，那么强制归一化相当于改坏了 kernel。

### 3.4 C-RLS 会精确拟合给定特征，特征变差会被忠实放大

C-RLS 的强项是：给定固定特征 $E$ 和标签 $Y$，它能高效、递归地得到解析解。

但它不会判断特征映射是否更好。若归一化后的 $\tilde{E}$ 降低了类间可分性，C-RLS 会忠实地在这个更差的空间里求最优解。

所以：

```text
C-RLS 保证的是固定特征空间中的求解质量
不保证任何特征变换都提升最终 mIoU
```

### 3.5 gamma 扫描没有生效，是因为当前尺度下 gamma 相对太小

第二组 `l2_sqrt gamma=0.1/1/10` 几乎完全相同。日志确认参数已传入，因此不是脚本问题。

更合理的解释是：

$$
E^\top E \gg \gamma I
$$

当 $E^\top E$ 的数值规模远大于 $\gamma I$ 时：

$$
E^\top E + 0.1I \approx E^\top E + 1I \approx E^\top E + 10I
$$

`l2_sqrt` 下每个像素特征范数约为：

$$
\sqrt{8196} \approx 90.53
$$

每个像素的平方范数约为 8196，而训练集中像素数量巨大，累积的 $E^\top E$ 会非常大。因此 `gamma=0.1/1/10` 都只是很小的扰动。

### 3.6 结论：归一化没有击中 RHL 的主要矛盾

RHL 的主要矛盾不是：

```text
row norm 是否一致
```

而是：

```text
随机高维基函数是否足够多样、稳定、与类别边界相关
```

因此，后续 RHL 改进应该从以下方向切入：

1. 降低随机特征采样方差。
2. 扩大或融合多个随机子空间。
3. 给随机特征加入类别或原型结构。

## 4. 方案一：RHL Subspace Ensemble, RHL-SE

### 4.1 方案定位

这是从 `6-15 CFSSeg_2D训练机制与轻量集成方案.md` 第 10 节“方案 A：RHL 多随机种子集成”中提取并优化后的版本。

推荐作为第一优先级，因为它同时满足：

| 条件 | 说明 |
|---|---|
| 贴合 RHL | 集成对象就是 RHL 随机子空间 |
| 贴合结项指标 | 可明确表述为集成学习与集成系统 |
| 工程成本低 | 不训练多个完整 DeepLab backbone |
| 理论合理 | 随机特征集成降低 Monte Carlo 方差 |
| 风险低 | 最差通常接近单模型，不应严重下降 |

### 4.2 对原方案 A 的审查与调整

原方案 A 的主思路是正确的：

```text
固定同一个 step0 backbone
只改变 RHL random seed
得到多个解析分类头
推理时融合概率
```

但需要做 4 个关键调整。

#### 调整 1：不要直接复用全局 `--random_seed`

当前 `train.py` 中 `--random_seed` 会影响：

```text
torch / cuda seed
numpy seed
python random seed
dataloader shuffle
随机数据增强
RandomBuffer 初始化
RecursiveLinear 新类别 tail 初始化
```

如果直接改 `--random_seed`，实验差异就不只是 RHL 子空间差异，还混入了 dataloader 顺序和训练增强随机性。

更干净的做法是新增：

```text
--rhl_seed
```

然后：

```text
random_seed 固定为 1
rhl_seed 使用 1/2/3/4/5
```

这样单模型差异主要来自 RHL 随机矩阵。

#### 调整 2：第一版不要叠加 `l2_sqrt`

归一化实验已经说明 `l2_sqrt_bs64` 不优于 `none_bs64`。所以 RHL-SE 第一版建议：

```text
RHL_NORM=none
```

先证明“多随机子空间集成”本身是否有效。

若 RHL-SE 成立，再尝试：

```text
gaussian none
BOA-RHL
PGH-RHL
```

不要第一轮把 ensemble 和 normalization 混在一起。

#### 调整 3：评估时先做 K=3，再做 K=5

推荐顺序：

| 阶段 | 实验 | 目的 |
|---|---|---|
| A0 | single seed 1/2/3 | 看不同 RHL seed 的方差 |
| A1 | ensemble K=3 | 看最小集成是否涨点 |
| A2 | ensemble K=5 | 如果 K=3 有收益，再扩展 |

如果单 seed 之间差异很小，ensemble 涨点也会有限，但仍可作为“稳定化系统”。

#### 调整 4：概率融合优先于硬投票

当前代码 `do_evaluate()` 使用：

```python
outputs = self.model(images)
outputs = torch.sigmoid(outputs)  # bce_loss
outputs = outputs.permute(0, 3, 1, 2)
outputs = F.interpolate(outputs, labels.shape[-2:], mode='bilinear')
preds = outputs.max(dim=1)[1]
```

因此 ensemble 应保持同一协议：

$$
p_{\text{ens}}(x) = \frac{1}{K}\sum_{k=1}^{K}\sigma(z_k(x))
$$

然后：

$$
\hat{y}(x) = \arg\max_c p_{\text{ens},c}(x)
$$

### 4.3 理论依据

单个 RHL 是对某个随机特征空间的一次采样：

$$
E^{(k)} = \operatorname{ReLU}(F\Phi_E^{(k)})
$$

不同随机种子得到不同 $\Phi_E^{(k)}$，也就是不同随机子空间。每个子空间通过 C-RLS 得到一个解析头：

$$
\hat{\Phi}^{(k)}
=
\left((E^{(k)})^\top E^{(k)}+\gamma I\right)^{-1}
(E^{(k)})^\top Y
$$

推理时融合：

$$
p_{\text{ens}}(x)
=
\frac{1}{K}\sum_{k=1}^K
\sigma\left(
\operatorname{RHL}^{(k)}(f(x))\hat{\Phi}^{(k)}
\right)
$$

直觉上，单个随机 RHL 可能对某些类别不友好。多个 RHL 从不同随机子空间观察同一个 dense feature，错误不会完全一致，平均后能降低方差。

这与 random feature 方法中的 Monte Carlo 方差降低一致：

$$
\operatorname{Var}\left[\frac{1}{K}\sum_{k=1}^{K}h_k(x)\right]
\approx
\frac{1}{K}\operatorname{Var}[h(x)]
$$

## 5. 方案二：Block-Orthogonal Antithetic RHL, BOA-RHL

### 5.1 动机

原始 RHL 使用随机矩阵。随机矩阵简单有效，但存在两个问题：

1. **随机特征冗余**：大量随机维度可能高度相关，浪费 buffer 容量。
2. **ReLU 半空间偏置**：`ReLU(fW)` 只保留正半轴响应，若某些有用方向落在负半轴，会被截断。

RHL 消融证明“高维随机非线性基函数”很重要。BOA-RHL 的目标不是调尺度，而是直接提升这些随机基函数的质量。

核心改动：

```text
Gaussian random W
    -> block-orthogonal W
    -> antithetic pair [W, -W]
    -> ReLU([FW, F(-W)])
```

### 5.2 理论直觉

#### 5.2.1 正交随机特征降低冗余

随机特征可以看作对某个隐式 kernel 的 Monte Carlo 近似。独立 Gaussian 随机特征容易出现冗余方向。正交随机特征让投影方向覆盖更均匀，通常能降低随机近似方差。

对 RHL 来说，这意味着：

```text
同样 buffer_size 下，更少维度浪费，更稳定的高维表示
```

#### 5.2.2 反向成对采样减少 ReLU 截断偏置

对于任意方向 $w$，同时使用 $w$ 和 $-w$：

$$
z^+(f)=\operatorname{ReLU}(fw)
$$

$$
z^-(f)=\operatorname{ReLU}(-fw)
$$

这两个特征合起来能表达投影在正负两侧的响应。因为：

$$
|fw| = \operatorname{ReLU}(fw) + \operatorname{ReLU}(-fw)
$$

也就是说，反向成对 RHL 比单侧 ReLU 更少丢失方向信息。

### 5.3 数学形式

构造半个随机矩阵：

$$
W_h \in \mathbb{R}^{d/2 \times C}
$$

然后拼接：

$$
W_{\text{anti}} =
\begin{bmatrix}
W_h \\
-W_h
\end{bmatrix}
$$

RHL 输出：

$$
E =
\operatorname{ReLU}(F W_{\text{anti}}^\top)
$$

若再使用 block-orthogonal 初始化，则 $W_h$ 由多个正交块拼接或裁剪得到。

### 5.4 代码实现方案

#### 5.4.1 新增参数

文件：`utils/parser.py`

```python
rhl_init: str = "gaussian"
```

argparse：

```python
parser.add_argument(
    "--rhl_init",
    type=str,
    default=Config.rhl_init,
    choices=["gaussian", "orthogonal", "orthogonal_antithetic"],
    help="Initialization strategy for RandomBuffer."
)
```

#### 5.4.2 修改 `RandomBuffer`

文件：`network/Buffer.py`

新增参数：

```python
rhl_init: str = "gaussian"
```

实现函数：

```python
def _init_orthogonal_blocks(self, rows, cols, device, dtype):
    blocks = []
    remain = rows
    while remain > 0:
        q, _ = torch.linalg.qr(torch.randn(cols, cols, device=device, dtype=dtype))
        block = q[:min(cols, remain), :]
        blocks.append(block)
        remain -= block.shape[0]
    return torch.cat(blocks, dim=0)
```

初始化逻辑：

```python
if rhl_init == "gaussian":
    self.reset_parameters()
elif rhl_init == "orthogonal":
    W = self._init_orthogonal_blocks(out_features, in_features, device, dtype)
    self.weight.copy_(W * scale)
elif rhl_init == "orthogonal_antithetic":
    half = out_features // 2
    W_half = self._init_orthogonal_blocks(half, in_features, device, dtype)
    W = torch.cat([W_half, -W_half], dim=0)
    if W.shape[0] < out_features:
        W = torch.cat([W, W_half[:1]], dim=0)
    self.weight.copy_(W[:out_features] * scale)
```

scale 建议先用：

```python
scale = math.sqrt(2.0 / in_features)
```

这与 ReLU/Kaiming 的量级接近。

#### 5.4.3 保持 C-RLS 不变

BOA-RHL 只改变固定映射 $F \mapsto E$，不改变：

```text
RecursiveLinear.fit()
RecursiveLinear.update()
ridge objective
```

因此仍然保持 CFSSeg 的闭式解主线。

## 6. 方案三：Prototype-Guided Hybrid RHL, PGH-RHL

### 6.1 动机

原始 RHL 完全随机，不知道类别结构。它能提升可塑性，但不保证随机维度对类别边界最有帮助。

语义分割中，每个类别在 frozen feature 空间里通常会形成一些原型区域。例如：

```text
person 像素的 dense feature 往往聚在某些区域
car / train / sofa 也会形成各自的特征簇
```

如果只用随机 RHL，分类头需要自己从随机特征里恢复这些类别结构。PGH-RHL 的想法是：

```text
保留原始随机 RHL
额外加入少量类别原型相似度特征
让 C-RLS 同时看到“随机高维特征”和“类别语义锚点”
```

这比归一化更贴近 RHL 的本质，因为它改变的是高维特征空间的内容，而不是只改变尺度。

### 6.2 数学形式

先用冻结 backbone 提取像素特征 $f_i$。

对每个类别 $c$ 计算一个或多个原型：

$$
\mu_c = \frac{1}{|\mathcal{I}_c|}
\sum_{i \in \mathcal{I}_c} f_i
$$

其中 $\mathcal{I}_c$ 是类别 $c$ 的像素集合。

构造 RBF 原型特征：

$$
g_c(f_i)
=
\exp\left(
-\frac{\|f_i-\mu_c\|_2^2}{2\sigma_c^2}
\right)
$$

最后拼接随机 RHL 与原型特征：

$$
E_i^{hybrid}
=
\left[
\operatorname{ReLU}(f_i W_r),
\lambda g_1(f_i),
\lambda g_2(f_i),
\cdots,
\lambda g_C(f_i)
\right]
$$

然后仍然用 C-RLS：

$$
\hat{\Phi}
=
\left((E^{hybrid})^\top E^{hybrid}+\gamma I\right)^{-1}
(E^{hybrid})^\top Y
$$

### 6.3 理论依据

PGH-RHL 可以理解为：

```text
Random feature expansion + prototype kernel features
```

它兼顾两种信息：

| 特征类型 | 作用 |
|---|---|
| 随机 RHL | 提供广泛非线性基函数，提高可塑性 |
| 原型 RBF | 提供类别相关局部相似度，强化类间边界 |

它仍然不引入可训练参数。原型是从数据统计得到的固定中心，类似 CFSSeg 中保存 $\Psi$ 这样的统计量。

### 6.4 代码实现方案

PGH-RHL 工程量比前两个方案更大，建议作为第二阶段方案。

#### 6.4.1 新增 Buffer 类

文件：`network/Buffer.py`

已有 `GaussianKernel` 类可复用思想，但它当前只返回 RBF 特征。建议新增：

```python
class PrototypeBuffer(Buffer):
    def __init__(self, centers, sigma=1.0, scale=1.0, device=None, dtype=torch.double):
        ...

    @torch.no_grad()
    def forward(self, X):
        # X: [B, HW, C]
        dist2 = torch.cdist(X.to(self.centers), self.centers).square()
        return torch.exp(-dist2 / (2 * self.sigma ** 2)) * self.scale
```

再新增混合 buffer：

```python
class HybridRandomPrototypeBuffer(Buffer):
    def __init__(self, random_buffer, prototype_buffer):
        ...

    @torch.no_grad()
    def forward(self, X):
        return torch.cat([self.random_buffer(X), self.prototype_buffer(X)], dim=-1)
```

#### 6.4.2 原型收集

文件：`trainer/trainer.py`

需要在 step1 的 C-RLS fit 前增加原型收集阶段。

推荐第一版只在 sequential `15-5` 上做，因为标签完整，不涉及伪标签噪声：

```python
@torch.no_grad()
def collect_class_prototypes(self, backbone, loaders):
    sums = {}
    counts = {}
    for loader in loaders:
        for X, y, _ in loader:
            feat, _ = backbone(X)
            feat = feat.view(B, C, -1).permute(0, 2, 1)
            y_small = F.interpolate(y.unsqueeze(1).float(), size=(H, W), mode="nearest").long()
            for cls in valid_classes:
                mask = (y_small == cls)
                sums[cls] += feat[mask].sum(dim=0)
                counts[cls] += mask.sum()
    centers = torch.stack([sums[c] / counts[c].clamp_min(1) for c in classes])
    return centers
```

使用 loaders：

```text
step0 train_loader0 + step1 train_loader
```

对 `15-5 sequential`，这能得到 0-20 全类原型。

#### 6.4.3 修改 AIR 输入维度

如果随机 RHL 是 8196 维，原型特征是 21 维，则：

```text
analytic_linear input dim = 8196 + 21
```

因此 `AIR.__init__()` 中：

```python
self.buffer_size = buffer_size + num_prototypes
self.analytic_linear = RecursiveLinear(self.buffer_size, gamma, ...)
```

或更干净地让 buffer 暴露：

```python
self.buffer.out_features
```

#### 6.4.4 保存原型

原型需要作为模型 buffer 保存进 checkpoint：

```python
self.register_buffer("centers", centers)
```

这样推理时可复现。

## 9. 最终建议

RHL 仍然是突破口，但应该从“归一化 RHL 输出”转向“改造 RHL 随机特征空间”。

当前最合理的路线是：

1. **先做 RHL-SE**：成本最低，符合集成系统指标，能自然利用 RHL 随机性。
2. **再做 BOA-RHL**：从随机特征质量层面改造 RHL，理论动机强于归一化。
3. **最后考虑 PGH-RHL**：如果需要更强论文贡献，再引入类别原型语义锚点。

一句话：

```text
RHL 归一化失败并不说明 RHL 不值得改；
它说明“只调尺度”太弱。下一步应该围绕随机子空间、随机特征方差和类别结构来改。
```
