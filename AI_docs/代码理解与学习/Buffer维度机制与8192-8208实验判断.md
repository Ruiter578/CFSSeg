# Buffer 维度机制与 8192 到 8208 实验判断

生成时间：2026-07-04  
项目路径：`/TRS-SAS/linwei/SegACIL`

## 0. 先给结论

在 SegACIL 里，`buffer` 不是数据缓存，也不是 replay memory。它指的是 RHL（Randomly-initialized Hidden Layer，随机初始化隐藏层）的输出维度，也就是代码里的 `RandomBuffer.out_features`。通俗地说：DeepLab 先把每个像素变成一个 256 维特征，`RandomBuffer` 再把这个 256 维特征随机扩展到 `buffer` 维，然后 `RecursiveLinear` 在这个高维空间里用闭式解更新分类器。

你的这组实验是在同一个 step0 权重 `20260621_baseline_bs16_16_trs` 的基础上，只扫描 step1 的 AIR/RHL 维度。可比性较强的 manifest 显示：`8192, 8196, 8204, 8208, 8212, 8216` 都使用同一个 base checkpoint hash、同一个 `batch_size=32`、`gamma=1`、`random_seed=1`、`rhl_seed=-1`、`rhl_norm=none`、`AIR feature=decoder`。其中 `8208` 的 overall `Mean IoU` 最高。

但这还不足以把原论文默认的 `8192` 全局改成 `8208`。更稳妥的表述是：

- 原论文默认值：`8192`。
- 当前本地 TRS、VOC `15-5 sequential`、固定 step0、step1 batch 32 的经验最优候选：`8208`。
- 若写论文或正式报告，应把 `8208` 称为“本地协议下调参得到的 tuned buffer size”，不要直接说“原论文应该更新为 8208”。

Codex 不能靠静态读代码、完全不跑实验，就搜索出真正最优的 `buffer`。它可以做的是：读代码找出合法搜索空间、整理已有结果、检查 manifest 可比性、估算显存/计算成本、生成搜索脚本、设计更严谨的实验网格。真正的 mIoU 最优点必须通过数据、特征、闭式求解和评估产生。

## 1. 原论文里的 buffer 是什么

论文在 Regression-Based Realignment 阶段引入 RHL。原始 dense feature 不直接接分类器，而是先通过一个随机初始化隐藏层扩展到更高维。论文把这个“扩展后的特征维度”叫作 buffer size。

对应关系可以理解为：

```text
原始像素特征 X:       [B, Hf*Wf, C]
RHL 随机扩展后 Z:     [B, Hf*Wf, buffer_size]
解析分类器输出 logits: [B, Hf, Wf, num_classes]
```

论文实现细节中说，continual learning 阶段设定 `buffer size = 8192`、`gamma = 1`，RHL 激活函数使用 ReLU。论文的 ablation 还说明：更大的 buffer 通常能提高 mIoU，但也带来更高计算成本，因此需要权衡。

参考来源：

- SegACIL arXiv HTML：<https://arxiv.org/html/2412.10834v1>
- 论文 III-B：RHL 将特征扩展到高维，并把 expanded feature dimension 称为 buffer size。
- 论文 IV Implementation Details：continual learning 阶段使用 `buffer size=8192`、`gamma=1`、ReLU。
- 论文 IV-B Buffer Size Analysis：更大 buffer 可能提升 mIoU，但计算成本也更高。

## 2. 代码里 buffer 在哪里

核心路径如下：

```text
run_trs.sh
  -> train.py
    -> Trainer(opts)
      -> step0: 普通 DeepLab 训练，不使用 AIR buffer
      -> step1: 加载 step0 checkpoint，构造 AIR
          AIR = frozen DeepLab backbone/classifier feature extractor
                + RandomBuffer/RHL
                + RecursiveLinear/C-RLS
```

关键代码位置：

| 文件 | 代码事实 |
|---|---|
| `run_trs.sh:15-21` | 当前 TRS 脚本使用 `BASE_SUBPATH=20260621_baseline_bs16_16_trs`，并硬编码 `BUFFER=8204`。 |
| `run_trs.sh:33-35` | 当前脚本只跑 `START_STEP=1` 到 `END_STEP=1`。 |
| `run.sh:30` | 通用脚本默认 `BUFFER="${BUFFER:-8192}"`，可被环境变量覆盖。 |
| `utils/parser.py:57,128` | CLI 参数名是 `--buffer`，默认 dataclass 值是 `16384`，但 runner 通常会覆盖。 |
| `trainer/trainer.py:20-55` | `AIR.__init__()` 用 `buffer_size` 创建 `RandomBuffer(backbone_output, buffer_size)` 和 `RecursiveLinear(buffer_size, gamma)`。 |
| `trainer/trainer.py:65-98` | `feature_expansion()` 先调用 backbone 提取 dense feature，再调用 `self.buffer(X)`，最后 `fit()` 把标签重采样到特征图大小并送入 `analytic_linear.fit()`。 |
| `trainer/trainer.py:369-430` | step1 加载 step0 模型，构造 AIR，先用 step0 loader realign，再用 step1 loader 增量更新。 |
| `network/Buffer.py:39-75` | `RandomBuffer` 是一个固定随机线性映射，`weight` 和 `bias` 用 `register_buffer` 保存，不是可训练参数。 |
| `network/Buffer.py:96-122` | 前向过程是随机线性投影加 ReLU，再可选做 RHL 归一化。 |
| `network/AnalyticLinear.py:90-165` | `RecursiveLinear` 保存 `R` 矩阵，并用闭式递推更新 `R` 和 `weight`。 |

这里有一个容易混淆的点：PyTorch 里的 `register_buffer` 只是说这个 tensor 跟着模型保存、加载、搬 GPU，但不是参数，不参与梯度训练。SegACIL 论文里的 `buffer size` 则是这个固定随机隐藏层的输出维度。两者都叫 buffer，但含义不完全一样。

## 3. buffer 在训练流程什么阶段发挥作用

### 3.1 step0：不发挥作用

`curr_step == 0` 时，代码走普通 DeepLab 训练：

```text
image -> DeepLab -> logits -> BCE loss -> loss.backward() -> optimizer.step()
```

这个阶段没有 `AIR`，也没有 `RandomBuffer`。所以你这次用 `20260621_baseline_bs16_16_trs` 的 step0 权重做基础时，buffer 不会改变 step0 模型本身。

### 3.2 step1 realign：第一次发挥作用

`curr_step == 1` 时，代码先加载 step0 checkpoint：

```text
checkpoints/20260621_baseline_bs16_16_trs/voc/15-5/sequential/step0/
  deeplabv3_resnet101_voc_15-5_step_0_sequential.pth
```

然后构造：

```text
AIR(
  backbone_output=256,
  backbone=step0_model,
  buffer_size=opts.buffer,
  gamma=opts.gamma,
  linear=RecursiveLinear,
)
```

接着用 step0 的训练数据跑一次 `self.model.fit(X, y)`，把原来的梯度训练分类头替换为解析分类器。这一步可以理解成“用 RHL + 闭式解重新对齐旧类”。

### 3.3 step1 当前数据：继续发挥作用

realign 后，代码继续用当前 step1 的训练数据调用：

```text
self.model.fit(X, y)
self.model.update()
```

这里 `RandomBuffer` 仍然是同一个固定随机映射。`RecursiveLinear` 会在同一个高维空间里递推更新旧类和新类的分类权重。对于 VOC `15-5`，step0 是 `0-15`，step1 是 `16-20`，所以这一步就是最终增量阶段。

### 3.4 step2 以后：如果有，就随 AIR checkpoint 继续用

在 `curr_step > 1` 时，代码加载上一步 `final.pth`，里面已经包含 AIR、RandomBuffer、RecursiveLinear 的状态。后续 step 会继续用同一个随机映射和 `R` 矩阵做递推更新。

## 4. buffer 在模型架构什么位置

以当前 `run_trs.sh` 的 `deeplabv3_resnet101` 为例：

```text
输入图像
  -> ResNet101 backbone
  -> DeepLabV3 ASPP + head_pre
  -> dense feature: [B, 256, Hf, Wf]
  -> 展平为每个像素一个特征: [B, Hf*Wf, 256]
  -> RandomBuffer/RHL: [B, Hf*Wf, buffer_size]
  -> RecursiveLinear: [B, Hf, Wf, num_classes]
  -> 上采样回标签大小
```

所以 `buffer` 位于“DeepLab 特征提取器之后、最终解析分类器之前”。它不是 backbone 的通道数，不是类别数，也不是 batch size。

当前代码在 step1 构造 AIR 时写死 `backbone_output=256`，所以 `buffer=8208` 的含义就是：

```text
每个像素的 256 维 DeepLab feature
  -> 随机投影 + ReLU
  -> 8208 维 RHL feature
```

## 5. 维度大小意味着什么

`buffer=D` 会同时改变几类张量的形状：

| 张量 | 形状 | 8192 到 8208 的变化 |
|---|---:|---|
| `RandomBuffer.weight` | `[D, 256]` | 随机投影方向多 16 个 |
| RHL 输出特征 | `[B, Hf*Wf, D]` | 每个像素多 16 个随机特征 |
| `RecursiveLinear.R` | `[D, D]` | 闭式递推的相关矩阵变大 |
| `RecursiveLinear.weight` | `[D, num_classes]` | 解析分类器输入维度变大 |

直观理解：

- 维度更大：随机特征空间更宽，线性分类器有更多“坐标轴”可以分开类别。
- 维度太小：表达能力可能不足。
- 维度太大：计算和显存明显增加，还可能带来数值条件、随机冗余、噪声拟合问题。

成本上，最重的是 `R` 矩阵。代码使用 `dtype=torch.double`，所以单个 `R` 大约是：

```text
D * D * 8 bytes
```

示例：

- `8192`：约 `512 MiB`。
- `8208`：约 `514 MiB`。

单看 8192 到 8208 只多约 2 MiB 的单矩阵存储，但代码里 `RecursiveLinear.fit()` 每批还会构造 `R_inv`、`S`、`S_inv` 等矩阵，并做 `D x D` 级别求逆，所以整体时间和临时显存不是只多 2 MiB。更大的 D，计算成本按二次到三次量级增长。

## 7. 8192 是否应该更新为 8208

分三个层级回答。

### 7.1 原论文默认值

不应该直接更新。原论文的 `8192` 是论文协议下的默认超参数，不能因为本地一次 sweep 就改写原论文事实。

### 7.2 你的本地实验默认值

可以把 `8208` 暂时作为当前 TRS、VOC `15-5 sequential`、step1 batch 32、固定 step0 权重的推荐候选。更准确的写法是：

```text
在 20260621_baseline_bs16_16_trs step0 checkpoint 上，
当前 buffer sweep 显示 8208 的 step1 all mIoU 最高。
```

如果后续继续沿用这条协议跑对照实验，`8208` 是合理的工程选择。

### 7.3 论文或正式方法默认值

暂时不建议把 `8208` 定为最终默认。至少还需要：

1. 对 top candidates 做重复验证：例如 `8196, 8204, 8208, 8212`。
2. 分离 `random_seed` 和 `rhl_seed`：只改 RHL 随机映射时，用多个 `rhl_seed` 看 8208 是否稳定。
3. 联动检查 `gamma`：buffer 改变特征维度和能量尺度，`gamma=1` 未必仍是局部最优。
4. 如果要写论文结论，不要只用 `test_results` 选超参。最好明确验证集选择策略，最终测试只报一次。
5. 如果目标不是只服务 VOC `15-5 sequential`，还要在 `15-1`、`disjoint`、`overlap` 或新 backbone 上复查。

## 8. 更好的搜索优化方式

推荐按“先省钱，再严谨”的顺序做。

### 8.1 固定可比性

每个实验必须固定：

```text
MODEL
AIR_FEATURE_SOURCE
BASE_SUBPATH
base checkpoint hash
task / setting / curr_step
batch size
gamma
random_seed
rhl_seed
rhl_norm
代码 commit
```

否则 buffer 变化和其他变量混在一起，结论会失效。

### 8.2 先做局部细扫

你现在已经看到 8208 附近有峰值，可以做更细的局部搜索：

```text
8184, 8192, 8200, 8204, 8208, 8212, 8216, 8224, 8232
```

如果成本允许，再加入：

```text
8064, 8128, 8256, 8320, 8448
```

这样既保留局部精度，又能确认 8208 是孤立峰还是更宽平台的一部分。

### 8.3 对 top candidates 做多 seed

建议不要只用 `random_seed=1, rhl_seed=-1`。更干净的做法是：

```text
random_seed 固定为 1
rhl_seed = 1, 2, 3, 4, 5
buffer = 8196, 8204, 8208, 8212
```

这样可以回答一个关键问题：8208 好，是因为维度真的更合适，还是因为这个维度下随机投影刚好幸运？

### 8.4 联动 gamma

论文说 `gamma` 比较鲁棒，但你的 sweep 是更细的 buffer 局部搜索。建议只做小规模联动：

```text
buffer = 8196, 8208, 8212
gamma = 0.1, 1, 10
```

如果 `8208 + gamma=1` 仍然领先，结论会稳很多。

### 8.5 让 runner 支持环境变量覆盖

当前 `run_trs.sh` 里是：

```bash
BUFFER=8204
```

这会导致 `BUFFER=8208 bash run_trs.sh` 无法覆盖。后续批量搜索更建议改成：

```bash
BUFFER="${BUFFER:-8208}"
```

这样每个实验可以用环境变量控制，不必每次手改脚本。注意：这是 runner 易用性改动，不改变 SegACIL 模型机制。

## 9. Codex 能不能静态、无需实验地搜索最优 buffer

不能。原因很具体：

- `buffer` 决定随机投影矩阵的形状。
- RHL 输出取决于真实图像经过 DeepLab 后的 dense feature。
- `RecursiveLinear.fit()` 的闭式解取决于真实特征、真实标签、ignore mask、类别分布和矩阵求逆。
- 最终 mIoU 取决于模型在验证/测试数据上的预测。

这些都不是静态代码里能推出的数值。代码只能告诉我们“怎么计算”，不能告诉我们“没跑过的 D 最后 mIoU 是多少”。

Codex 可以做的静态或半静态工作包括：

- 读取代码，确认 `buffer` 的真实作用点。
- 检查实验结果是否同源、同 base checkpoint、同 seed、同 batch size。
- 汇总已有 `test_results_*.json` 和 `run_manifest.json`。
- 估算 `D x D` 矩阵显存和计算量。
- 发现 runner 是否硬编码，是否会破坏批量搜索。
- 生成搜索脚本、结果汇总脚本和 Markdown 报告。
- 做低成本 proxy，比如只跑少量 batch，统计 RHL 特征范数、rank、条件数、类别 separability。但这已经不是“纯静态”，而且 proxy 不能替代最终 mIoU。

所以最准确的边界是：

```text
Codex 可以帮你把搜索做得更严谨、更省钱、更不容易混变量；
但不能不用实验就凭静态分析决定 8208 一定是全局最优。
```

## 11. 一句话记忆

SegACIL 的 `buffer` 可以理解为“每个像素随机扩展后的高维特征长度”。它在 step1 及后续 AIR 解析增量阶段发挥作用，位置在 DeepLab dense feature 和 RecursiveLinear 分类器之间。8208 目前是你这组本地实验的好候选，不是已经被证明的论文级新默认值。
