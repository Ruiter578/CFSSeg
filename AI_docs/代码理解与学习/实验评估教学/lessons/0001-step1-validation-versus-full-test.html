Step1 验证子集与完整测试集

目标：看到一个 mIoU 数字时，先问“它在哪些图像、哪些类别上算出来的？”

这次代码实际做了什么

完整 test：VOC val.txt 的 1,449 张图 -> 覆盖完整 21 类 step1 val：从 val_cls.txt 挑出“含本步新类 16--20”的图 -> 354 张图 -> 不是完整 21 类的代表性评估集

筛选规则在 utils/tasks.py。因此，aeroplane、boat、cow、horse 在这 354 张图中各出现 0 次。

为什么结果仍显示 IoU=0

指标实现没有把“GT 中不存在”的类别排除，而是在分母加 EPS=1e-6。于是 0/eps 仍是 0，而不是 NaN；随后 Mean IoU 又把这个 0 平均了进去。代码见 stream_metrics.py。

所以该 0 不能解读为“模型忘了该类别”。它首先意味着：该类别在当前评估样本中不可被召回率衡量。

正确的决策分工

训练集：拟合参数 验证集：在不看完整 test 的前提下，淘汰候选 完整 test：配置冻结后，报告泛化性能

当前仓库没有天然的“完整 21 类独立 validation split”。因此 354 张 step1 val 可以作为增量子集诊断，但不能作为完整 mIoU 的唯一排序依据。完整 test 审计可以说明结果，但若反复据它挑参数，也会造成 test-selection bias。

快速检验

问题：某类别在评估 GT 中没有像素，而代码使用 TP / (GT + Pred - TP + EPS)。这个类别的 IoU=0 能否证明模型没有预测该类？
