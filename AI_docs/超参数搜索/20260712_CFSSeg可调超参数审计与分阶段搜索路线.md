# CFSSeg / SegACIL 可调超参数审计与分阶段搜索路线

> 生成日期：2026-07-12  
> 代码范围：`/TRS-SAS/linwei/SegACIL` 当前 2D VOC 主流程  
> 论文依据：[CFSSeg arXiv v2](https://arxiv.org/abs/2412.10834)、[HTML 正文](https://arxiv.org/html/2412.10834)  
> 当前推荐基线：`buffer=8224, gamma=1, rhl_norm=none`

## 1. 结论先行

除了已经完成的 buffer 搜索，当前最值得继续系统优化的变量不是简单地把
Shell 中所有参数逐个扫一遍，而是以下四条主线：

1. **闭式阶段的特征与正则**：`air_feature_source`、`gamma`、增量 batch。
   它们可以复用 step0 权重，成本相对低，应最先做。
2. **实现级随机先验**：`RecursiveLinear.fit()` 中新类别权重的
   `epsilon=1e-3`。它不是论文显式超参数，却会改变岭回归解，值得独立消融。
3. **step0 表征质量**：backbone/classifier 分组学习率、loss、step0 batch、
   weight decay。后续 encoder 被冻结，所以 step0 表征决定整个 CFSSeg 上限。
4. **结构与训练策略**：output stride、BN 策略、训练时长和 LR schedule。
   这些可能影响较大，但成本高、交互多，应放在前两轮之后。

推荐的实际执行顺序是：

```text
实验正确性门禁
-> AIR feature source
-> analytic tail epsilon
-> gamma
-> 增量 batch / 数值与效率
-> step0 分组学习率
-> BCE vs CE
-> step0 batch
-> weight decay
-> output stride / BN / schedule
-> 跨 task 与跨 setting 确认
```

不建议搜索 `method`、`val_interval`、`print_interval`、`pin_memory`、
`rhl_seed` 的“最佳值”、测试裁剪方式或任务划分。这些参数要么当前无效，
要么只影响速度/日志/复现性，要么属于评测协议而不是可合法优化的超参数。

## 2. 论文事实与代码事实

### 2.1 CFSSeg 原论文的 2D 配置

CFSSeg 使用 VOC2012、DeepLabv3 + ImageNet 预训练 ResNet-101。step0 训练
50 epochs、batch 32、SGD、学习率 `1e-2`、momentum `0.9`、weight decay
`1e-4`、poly schedule 和 BCE。增量阶段冻结 encoder，使用 RHL + C-RLS，
论文给出 `d_E=8192`、`gamma=1`、不确定性阈值 `tau=0.4`。原文还强调
增量阶段只单遍处理当前数据。[论文实验设置](https://arxiv.org/html/2412.10834#S5.SS1)

论文消融证明 RHL 和 pseudo-labeling 都重要，但没有证明论文给出的所有数值
都是全局最优。特别是 `8192`、`gamma=1` 和阈值都属于经验设置，继续做受控
搜索在方法学上是合理的。

### 2.2 当前代码的实际主链路

```text
step0:
图像 -> DeepLabv3/ResNet101 -> BCE/CE -> SGD + scheduler -> step0 checkpoint

step1+:
冻结 step0 模型 -> 选择 decoder/aspp 特征 -> RandomBuffer/RHL
-> RecursiveLinear/C-RLS -> 测试
```

关键代码位置：

- 启动参数：`run.sh:9-151`
- parser：`utils/parser.py:5-190`
- step0 optimizer：`trainer/trainer.py:498-506`
- step0 训练：`trainer/trainer.py:509-559`
- AIR 构造：`trainer/trainer.py:560-623`
- AIR 特征接口：`network/_deeplab.py:126-187`

## 3. 对抗性代码审查：搜索前必须处理的问题

### 3.1 Critical：Shell 的 `LR` 当前没有真正控制 optimizer 学习率

`run.sh` 会传 `--lr "$LR"`，parser 也会得到 `opts.lr`。但是
`trainer/trainer.py:498-505` 为两个参数组显式写死：

```python
training_params = [
    {'params': self.model.backbone.parameters(), 'lr': 0.001},
    {'params': self.model.classifier.parameters(), 'lr': 0.01},
]
optimizer = torch.optim.SGD(training_params, lr=self.opts.lr, ...)
```

PyTorch 参数组已经有自己的 `lr` 时，optimizer 的全局 `lr` 不会覆盖它们。
因此只在 Shell 中扫描 `LR=0.005/0.01/0.02`，实际三组仍会使用
`0.001/0.01`，属于**伪搜索**。

搜索前必须显式增加：

```text
BACKBONE_LR
CLASSIFIER_LR
```

或者增加 `CLASSIFIER_LR` 与 `BACKBONE_LR_MULTIPLIER`，并把真实解析值写入
`run_manifest.json`。

### 3.2 Warning：当前 `focal_loss` 等价于 CE

`utils/loss.py:5-21` 中 `FocalLoss(gamma=0)`，构造 criterion 时没有覆盖
gamma。因为 `(1-pt)^0 = 1`，当前 focal loss 数学上退化成普通 CE。

所以当前只值得比较：

```text
bce_loss vs ce_loss
```

除非先把 `FOCAL_GAMMA` 暴露出来，否则不要再跑 `focal_loss`，它只会重复 CE。
真正的 focal gamma 搜索属于类别不平衡方法实验，不应伪装成已有 loss 三选一。

### 3.3 Critical：不能继续使用最终 test 结果反复选超参数

step1 之后代码直接调用 `do_evaluate(mode='test')`，并把结果保存为
`test_results_*.json`（`trainer/trainer.py:623,648,650-697`）。此前 buffer
搜索正是从这些 test JSON 选择最优值。

如果对同一个 test 集反复试几十组并选择最高值，最高结果会带有选择偏差。
后续搜索必须：

1. 使用内部 validation 结果进行排序和淘汰；
2. 配置锁定后，再运行一次官方 test；
3. 不根据 test 结果返回上一阶段重新调参。

在实现层面，应让增量阶段保存独立的 `val_results_*.json`，队列只读取 val。

### 3.4 Warning：新类别分类权重存在论文未说明的随机先验

`network/AnalyticLinear.py:135-139` 在出现新类别时，用随机小量扩展分类器：

```python
epsilon = 1e-3
tail = torch.randn(...) * epsilon
self.weight = torch.cat((self.weight, tail), dim=1)
```

对带正则的 RLS 来说，初始权重不会完全消失。单次更新可写成近似形式：

```text
W = (X^T X + gamma I)^-1 (X^T Y + gamma W0)
```

因此随机 `W0` 相当于一个非零随机 ridge prior。论文标准岭回归公式是零中心
正则，没有给出这个 `1e-3`。该值可能直接影响新类别 mIoU，也会让全局
`random_seed` 除了控制数据顺序外还控制新类别权重先验。

建议把它暴露为 `ANALYTIC_TAIL_EPSILON`，独立比较 `{0, 1e-4, 1e-3}`。
这是实现语义消融，不是普通训练技巧；若 `0` 更好，也更贴近论文闭式解。

### 3.5 Warning：buffer=8224 的结论是条件最优，不是全局常数

现有结果支持 `8224` 优于已测试邻域，并优于论文 `8192` 的当前复现结果。
但它依赖以下条件：

- 当前 step0 checkpoint；
- `air_feature_source=decoder`；
- `gamma=1`；
- `rhl_norm=none`；
- 当前 RHL 初始化和 C-RLS 实现。

如果 feature source、step0 表征、output stride 或 tail epsilon 改变，特征尺度
和相关矩阵条件数也会改变，buffer 最优点可能移动。因此新阶段只有在上述关键
条件发生变化时，才做 `{8208, 8224, 8240}` 的局部复核，不重新全范围盲扫。

同时应克制“8224 远好于 8192”的表述：当前证据说明其工程上更优，但单次
最佳值同时包含 RHL seed 波动。严谨报告应比较同 seed 均值和标准差。

## 4. 候选参数总表

| 参数/变量 | 当前值 | 是否真实生效 | 是否重训 step0 | 预期影响 | 成本 | 优先级 |
|---|---:|---|---|---|---|---|
| AIR feature source | decoder(auto) | 是 | 否 | 高 | 低 | P1 |
| analytic tail epsilon | `1e-3` 硬编码 | 是 | 否 | 中-高，偏新类 | 低 | P1 |
| gamma | 1 | 是 | 否 | 中；论文可能较鲁棒 | 低 | P1 |
| 增量 batch size | 32 | 是 | 否 | 低-中；数值/速度 | 低 | P1 |
| backbone/classifier LR | `0.001/0.01` 硬编码 | 是；Shell LR 无效 | 是 | 高 | 高 | P2 |
| loss type | BCE | 是 | 是 | 高 | 高 | P2 |
| step0 batch size | 历史 checkpoint 多为 16；论文 32 | 是 | 是 | 中-高 | 高 | P2 |
| weight decay | `1e-4` | 是，但 Shell 未传 | 是 | 中 | 高 | P2 |
| output stride | 8 | 是 | 是 | 中-高、结构邻接 | 很高 | P2 |
| train epochs | 50 | step0 生效；step1 不生效 | 是 | 中 | 高 | P3 |
| LR policy | poly | 是 | 是 | 中 | 高 | P3 |
| BN freeze | false | 是，但 Shell 未传 | 是 | 中 | 高 | P3 |
| BN momentum | `0.01` 硬编码 | 是 | 是 | 低-中 | 高 | P3 |
| crop size | parser 默认 513 | 是，但 Shell 未暴露 | 是 | 中 | 很高 | P3 |
| augmentation scale | `[0.5, 2.0]` 硬编码 | 是 | 是 | 中 | 很高 | P3 |
| pseudo-label confidence | 0.7 | 仅非 sequential | 否 | 高，特定 setting | 中 | 条件 P1 |
| rhl_seed | -1/显式 seed | 是 | 否 | 方差评估 | 中 | 只做复现 |
| rhl_norm | none | 是 | 否 | 方法变量 | 中 | 不纳入 |

这里 P1/P2/P3 表示**建议执行顺序**，不是只按理论影响排序。P1 参数的优势是
复用现有 step0，能以较低代价先排除大量不确定性。

## 5. 按预期影响排序

若只看“可能改变最终结果的幅度”，不考虑实验成本，排序建议为：

1. step0 分组学习率；
2. step0 loss（BCE vs CE）；
3. AIR feature source；
4. output stride；
5. step0 batch 与 BN 统计；
6. analytic tail epsilon；
7. weight decay；
8. gamma；
9. 训练时长与 LR schedule；
10. crop/augmentation；
11. 增量 batch size。

但实际执行不能照这个顺序直接开始，因为前五项大多要重训昂贵的 step0。
应先做便宜 P1，再进入 step0 搜索。

## 6. 分阶段实验组

## G0：实验正确性门禁（不跑 GPU 搜索）

### G0.1 参数接线

在启动任何新 sweep 前完成：

- 暴露并记录 `BACKBONE_LR`、`CLASSIFIER_LR`；
- 暴露 `ANALYTIC_TAIL_EPSILON`；
- 若要研究 focal，再暴露 `FOCAL_GAMMA` 和 `FOCAL_ALPHA`；
- 把 optimizer 两组真实 LR、momentum、Nesterov、poly power 写入 manifest；
- 为增量阶段保存 validation JSON；
- 队列锁定 Git commit、tracked 状态、基础 checkpoint SHA256。

### G0.2 统一选择规则

所有实验统一：

- 主指标：validation All mIoU；
- 辅助指标：old mIoU、new mIoU、运行时间、峰值显存；
- 单 seed 初筛，Top-2 使用相同 seed 做配对确认；
- 至少用 3 个独立 `rhl_seed` 确认闭式阶段候选；
- `rhl_seed=-1` 不和 `rhl_seed=1` 重复计数；
- 实用改进阈值建议设为 All mIoU `+0.001`（0.1 个百分点）；
- 若综合指标接近，优先选择新类更高、方差更小、成本更低的配置；
- 最终 test 只在整套配置冻结后评估一次。

## E1：低成本闭式阶段搜索

固定：当前 step0 checkpoint、`buffer=8224`、`rhl_norm=none`、VOC 15-5
sequential、同一 commit。

### E1.1 AIR feature source（最高优先）

当前 DeepLabv3 支持 `decoder` 和 `aspp`，二者都是 256 通道，可以直接送入
现有 RHL，不必重训 step0。

| 阶段 | 取值 | seed | 晋级规则 |
|---|---|---|---|
| 初筛 | `decoder`, `aspp` | rhl_seed=2 | aspp 若未落后超过 0.001，进入确认 |
| 确认 | Top-2 | rhl_seed=2,4,5 | 配对均值、方差和 old/new Pareto |

如果 `aspp` 胜出，必须局部复核 buffer `{8208,8224,8240}`，因为特征分布改变。
如果 decoder 明确胜出，保留当前 buffer 结论，不重扫。

### E1.2 analytic tail epsilon

| 取值 | 含义 |
|---:|---|
| `0` | 标准零中心 ridge，新类权重从零开始 |
| `1e-4` | 很弱随机先验 |
| `1e-3` | 当前实现 |

先用 rhl_seed=2 初筛；Top-2 再用 4、5 配对。重点看新类 mIoU、跨 seed
方差和是否更接近论文的解析解。不要把 epsilon 和 gamma 同时改变。

### E1.3 gamma 粗搜与细搜

在 feature source 和 epsilon 固定后执行：

```text
粗搜：{0.01, 0.1, 1, 10, 100}
细搜：只在粗搜出现明确峰值时，在相邻数量级补 {0.03, 0.3, 3, 30}
```

粗搜用 rhl_seed=2；Top-2 用 `2,4,5` 确认。停止规则：若粗搜最大与最小
All mIoU 差小于 `0.001`，且没有 NaN/Inf/求逆异常，保留论文值 `gamma=1`，
不做细搜。gamma 应使用对数尺度，不要像 buffer 一样做线性步长搜索。

### E1.4 增量 batch size：以等价性和效率为目标

候选：`{16,32,64}`。batch 64 先做显存可行性检查；论文效率表使用 64，
本地历史 `buffer=8200,batch=64` 也曾成功，但 `buffer=8224` 仍需重新核验。

理论上闭式统计应对 batch 划分近似不敏感；当前实现逐 batch 求逆且含随机
新类初值，所以有限精度下仍可能变化。选择规则不是“挑偶然最高分”，而是：

```text
若 mIoU 差在 ±0.001 内，选择更快且不 OOM 的最大 batch；
若差异超过阈值，按相同 seed 重复，视为数值稳定性问题调查。
```

## E2：高收益但昂贵的 step0 搜索

每个 step0 候选都必须继续跑到 step1，并用最终 validation 指标排序。只比较
step0 自身初始类验证分数不够，因为 CFSSeg 更关心冻结特征对新类的可分性。

### E2.1 分组学习率

先修复 LR 接线，再做坐标式搜索，避免 3x3 全组合：

```text
阶段 A：classifier_lr ∈ {0.005, 0.01, 0.02}
        backbone_multiplier 固定 0.1

阶段 B：在最佳 classifier_lr 下
        backbone_multiplier ∈ {0.05, 0.1, 0.2}
```

共 5 个不重复配置。Top-2 用 global random_seed `{1,2,3}` 完整重训确认。
学习率的预期影响最高，因为后续 encoder 被永久冻结。

### E2.2 Loss

```text
bce_loss vs ce_loss
```

使用 E2.1 最佳 LR，每种 loss 完整训练 step0+step1。不要运行当前
`focal_loss`，因为它等价 CE。若未来研究真正 focal loss，至少比较
`focal_gamma={1,2}`，并把它归入“类别不平衡方法消融”，而非本轮干净搜索。

### E2.3 step0 batch size

```text
{16, 32}
```

16 对应当前主要基础 checkpoint，32 对应 CFSSeg 论文。batch 改变梯度噪声、
BN 统计、每 epoch iteration 数和 drop_last 样本，必须从头训练。若 32 胜出，
当前基于 bs16 step0 得出的 buffer=8224 需要做局部复核，而不是直接沿用。

### E2.4 weight decay

使用对数网格：

```text
{1e-5, 1e-4, 1e-3}
```

只有当相邻点出现明确趋势时，补 `3e-5` 或 `3e-4`。Top-2 再做多 global seed。
不要和 LR 同时网格化；两者都影响收敛，先固定 LR 再搜 weight decay。

## E3：结构邻接与训练策略

这组可能影响大，但会显著改变计算量或模型行为，只有 E1/E2 收敛后再做。

### E3.1 output stride

```text
{8,16}
```

stride 8 保留更高空间分辨率但更耗显存；stride 16 更快、更省显存但可能损失
边界和小目标。该变量改变 step0 网络膨胀卷积设置和增量阶段像素特征数量，
属于结构邻接实验。若胜出配置改变，需局部复核 buffer/gamma。

### E3.2 BN 策略

先比较：

```text
bn_freeze = false vs true
```

若不冻结胜出且方差较大，再研究硬编码 BN momentum `{0.01,0.1}`。不要同时
扫描 batch 和 BN；先选 batch，再比较 BN，否则无法解释收益来源。

### E3.3 训练时长与 LR schedule

建议顺序：

```text
epochs: {30,50,75}
-> 在最佳 epoch 下再比较 poly vs warm_poly
```

`step` scheduler 的 `step_size=10000` 与总 iteration 强耦合，不适合直接作为
类别选项；若要使用，必须先按总 iteration 重新设计衰减点。poly power 当前
硬编码 0.9，只有在 poly/warm_poly 都不理想时才暴露 `{0.7,0.9,1.0}`。

### E3.4 crop 与增强

最后才考虑：

```text
crop_size: {385,513}
random scale range: baseline [0.5,2.0] vs narrower [0.75,1.5]
```

513 是论文和 DeepLab 常用协议。改变 crop 会同时影响上下文、batch 可行性和
iteration 成本，不建议为了单次 test 分数随意加入 641 等超大裁剪。

## E4：协议特定参数与最终泛化确认

### E4.1 pseudo-label threshold 只用于 disjoint/overlap

sequential 有完整旧类标签，不应搜索伪标签参数。对于 CFSSeg 原始固定阈值：

```text
pseudo_label_confidence ∈ {0.5,0.6,0.7,0.8,0.9}
```

论文 `tau=0.4` 是不确定性阈值，对应代码 confidence 约 `0.6`。该搜索必须在
disjoint/overlap 中单独运行，不得和 sequential 超参数混合。当前仓库的
adaptive/classwise pseudo-label 属于方法创新分支，不纳入本轮纯超参数优化。

### E4.2 跨协议确认

最终只保留一套全局配置，按顺序确认：

1. VOC 15-5 sequential：用于搜索；
2. VOC 15-1 sequential：验证多 step 和类序稳定性；
3. VOC 15-1 disjoint 或 10-1 overlap：验证 semantic drift 场景；
4. 每个最终协议用 global random_seed `{1,2,3}` 报均值和标准差；
5. 最终配置锁定后才读取官方 test。

不建议为每个 task 分别挑一套最优超参数再汇报，这会放大调参自由度并削弱
“方法通用”的结论。

## 7. 不值得搜索或需要单独立项的变量

| 变量 | 判断 | 原因 |
|---|---|---|
| `method` | 不搜 | 当前没有真实方法分支，主要是记录字段 |
| `val_interval` | 不搜 | parser 存在，但训练逻辑没有按该值控制验证频率 |
| `print_interval` | 不搜 | 训练代码仍硬编码 `seq % 10`，且只影响日志 |
| `val_batch_size` | 不搜性能 | 只影响评估吞吐，预测应一致 |
| `pin_memory`/workers | 不搜性能 | 工程吞吐变量，不是模型质量参数 |
| `crop_val` | 不搜 | 改变评测协议，不能用于刷分 |
| `random_seed` | 不优化 | 用于估计方差，不应选择“最好 seed” |
| `rhl_seed` | 不优化 | 只做随机投影稳定性，不部署“最佳 seed” |
| `rhl_norm_eps` | 不搜 | `rhl_norm=none` 时完全无作用 |
| `rhl_norm` | 单独方法实验 | 此前已尝试并因收益不足搁置 |
| `separable_conv` | 当前 V3 不搜 | 代码只在模型名含 `plus` 时生效 |
| `pretrained_backbone` | 固定开启 | 关闭会改变基线前提，预期显著变差 |
| model/backbone | 单独架构实验 | DeepLabv3+、ResNet50/101 不是普通超参数 |
| RHL activation/bias | 单独方法消融 | 改变论文 RHL 定义，不属于干净调参 |
| dtype float/double | 数值/效率研究 | 当前 double 是求逆稳定性保障，不宜先动 |

## 8. 资源预算与队列建议

按“单 seed 初筛 + Top-2 三 seed 确认”估算：

| 阶段 | 预计初筛组数 | 确认组数 | 是否重训 step0 |
|---|---:|---:|---|
| E1.1 feature source | 1 个新增候选 | 最多 3 | 否 |
| E1.2 tail epsilon | 3 | 最多 6 | 否 |
| E1.3 gamma | 5，必要时 +4 | 最多 6 | 否 |
| E1.4 incremental batch | 3 | 异常时重复 | 否 |
| E2.1 LR | 5 | Top-2 x 3 global seeds | 是 |
| E2.2 loss | 2 | Top-2 x 3 | 是 |
| E2.3 step0 batch | 2 | Top-2 x 3 | 是 |
| E2.4 weight decay | 3 | Top-2 x 3 | 是 |

不要一次把所有队列都启动。每个阶段结束后根据停止规则缩小下一阶段：

```text
E1.1 结束 -> 决定是否复核 buffer
E1.2 结束 -> 固定解析先验
E1.3 结束 -> 固定 gamma
E1.4 结束 -> 固定效率 batch
E2.1 结束 -> 才生成 loss 队列
```

这样避免后续实验建立在已被淘汰的配置上。

## 9. 推荐的第一批下一步实验

在任何 GPU 队列之前，先完成 G0 的参数接线和 validation-only 汇总。之后第一批
只启动 E1.1：

```text
固定：
  base checkpoint = 当前已验证 step0
  buffer = 8224
  gamma = 1
  rhl_norm = none
  incremental batch = 32
  setting = sequential 15-5

实验：
  air_feature_source = aspp
  rhl_seed = 2,4,5

对照：
  已有 decoder / rhl_seed = 2,4,5
```

这是当前信息增益最高、成本最低、最容易解释的一组。只有它完成后，才决定
是否需要局部重验 buffer，再进入 tail epsilon 和 gamma。

## 10. 最终建议

1. 不要立即搜索 Shell 的 `LR`；当前它不生效，先修复参数接线。
2. 不要运行 `focal_loss`；当前它与 CE 重复。
3. 不要再用 test JSON 做中间淘汰；先增加 validation 结果输出。
4. 第一批搜索 AIR feature source，随后审查 tail epsilon，再做 gamma。
5. step0 参数按 LR -> loss -> batch -> weight decay 顺序做坐标搜索。
6. 每次关键特征分布改变后，只局部复核 buffer，不重新全范围扫描。
7. `8224` 作为当前条件最优值保留，但不宣称它对所有 checkpoint、feature
   source、task 和数据集都普遍最优。
8. 3D S3DIS/ScanNet 的 DGCNN、buffer=5000、dataset-specific threshold 是
   独立实验体系，不能与当前 VOC 2D 排名混为一谈；应在 2D 路线收敛且 3D
   数据/代码复现通过后另立搜索文档。
