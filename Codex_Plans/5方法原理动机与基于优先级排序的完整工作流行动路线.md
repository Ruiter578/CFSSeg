# 5 方法原理动机与基于优先级排序的完整工作流行动路线

> 日期：2026-06-20
> 目标任务：SegACIL，VOC 15-5 sequential step1
> 适用对象：方法理解、Codex 代码实现、实验编排与结果归因

## 0. 这五个方法在解决什么

当前 RHL 流程可简化为：backbone 提取每个像素的特征，固定随机映射把特征展开到高维空间，RecursiveLinear 用解析方式拟合类别输出。五个方法分别干预不同环节：

1. **P1 BOA-RHL**：减少随机映射方向的冗余；
2. **P2 PGH-RHL-lite**：给纯随机特征增加类别语义锚点；
3. **RHL 归一化新方案**：保留有用幅值的同时控制尺度，并修正类别不平衡的解析更新；
4. **P3 Snapshot**：从不同 backbone 状态产生真正结构不同的成员；
5. **RHL-SE 2.0**：在输出端逐像素判断哪个成员更可靠。

它们不是五个互斥替代品，也不是必须顺序串联的五步优化。其完整数据流为：

```text
image
  -> backbone(snapshot)
  -> feature F
  -> BOA random branch H_rand --------+
  -> PGH prototype branch H_proto ----+--> branch normalization
                                           -> C-RLS / weighted C-RLS
                                           -> member logits

multiple member logits
  -> calibration + pixel reliability
  -> RHL-SE 2.0
  -> final prediction
```

## 1. 科研小白版机制解释

### 1.1 BOA-RHL：让随机方向少重复

普通随机映射像是从许多随机角度观察同一个 feature。如果方向高度相似，8196 维看似很大，真实新增的信息却有限。orthogonal 初始化尝试让方向彼此分散，使有限维度覆盖更丰富的空间。

它主要解决：随机高维特征冗余。它不直接知道类别语义，也不解决类别不平衡。

### 1.2 PGH-RHL-lite：加入“像哪一类”的显式线索

纯随机特征没有“这像汽车/人/植物”的直接含义。PGH 从训练集提取每类一个或多个原型，然后计算当前像素与原型的 cosine 或 RBF 相似度，再与随机特征拼接。

它主要解决：随机特征缺少语义锚点。多原型、k-means 和 sigma 搜索分别用于表示类内多模态和控制 RBF 响应范围。

### 1.3 RHL 归一化新方案：控制幅值，并让新类在解析更新中被看见

完全归一化会把所有像素拉到同一范数，可能丢掉有用幅值；完全不管尺度又可能造成数值和分支失衡。PowerNorm 用连续参数控制压缩强度。

类别感知 C-RLS 则解决另一个问题：背景和旧类像素太多时，最小二乘更关心多数类。它通过样本权重提高稀缺类别对解析解的影响。

### 1.4 Snapshot：让成员差异来自 backbone，而不只是随机头

现有 RHL seeds 只改变随机映射，成员预测非常相似。Snapshot 使用 step0 不同 epoch 的 backbone checkpoint，再分别训练 step1 解析头，产生更深层的表示差异。

它主要解决：集成成员缺乏结构级多样性。它是系统级方案，成本高于只换随机矩阵。

### 1.5 RHL-SE 2.0：只在有证据时切换成员

现有 class-wise 权重只知道“某个类别通常信谁”，但一个类别内部的边界、遮挡和主体区域难度不同。RHL-SE 2.0 使用 margin、entropy、成员一致性、类别校准和 old/new prior，为每个像素估计成员可靠性。

它主要解决：已有成员的互补性无法被静态平均利用。它不能凭空创造单成员没有的能力，因此应在成员方法之外独立评价。

## 2. 方法边界与依赖矩阵

| 方法 | 作用点 | 训练新单成员 | 依赖 | 可自然组合 | 主要耦合风险 |
|---|---|---:|---|---|---|
| BOA-RHL | random matrix | 是 | RHL 基线 | PGH、PowerNorm、Snapshot、SE2 | scale/gamma、buffer 维度 |
| PGH-RHL | semantic feature branch | 是 | prototype bank | BOA、PowerNorm、CA-C-RLS、Snapshot、SE2 | 分支尺度、总维度、标签泄漏 |
| PowerNorm/CA-C-RLS | feature scale / analytic objective | 是 | RHL 或 hybrid feature | BOA、PGH、Snapshot、SE2 | 有效正则、类别权重混杂 |
| Snapshot | backbone checkpoint | 是，多成员 | step0 snapshot 基础设施 | 任一 step1 方法、SE2 | 成员质量/多样性、训练成本 |
| RHL-SE 2.0 | output routing | 否，训练 gate | 多个成员与 val calibration | 上述全部 | val 过拟合、异构配置混用 |

### 2.1 强依赖

- PGH 必须先有无泄漏、可复现的 prototype bank。
- Snapshot 必须先实现中间 checkpoint 保存和显式 `base_ckpt`。
- RHL-SE 2.0 必须先有 member manifest、validation 数据隔离和成员输出。

### 2.2 弱依赖/可并行

- BOA P1.5 与 PGH prototype bank 可以并行推进。
- RHL-SE 2.0 可以直接使用已有 seeds 1/2/3 开发，不必等 PGH/Snapshot。
- Snapshot 基础设施可以在单成员方法实验期间并行准备。

### 2.3 自然组合不等于默认融合

BOA+PGH 在同一 hidden feature 中自然拼接；PowerNorm 可作用于两分支；Snapshot 可为这些方法提供多个 backbone；SE2 可融合最终成员。但每多开一个模块，结果归因就增加一个交互项。正确流程是：先独立，后做 2x2 因子实验，最后做系统组合。

## 3. 优先级排序

### 优先级 1：BOA-RHL 1.5 复核

理由：已有唯一正向单模型信号，现有代码已实现，新增成本主要是严谨复核和诊断。先回答 orthogonal 的收益能否跨 seed、是否只是 scale，以及 antithetic 增大 BUFFER 后能否恢复。

必须完成：

- orthogonal/legacy seeds 1/2/3 paired 对照；
- scale 与 gamma 匹配；
- BUFFER 8196/12288/16392 的资源 smoke 和公平比较；
- effective rank/coherence/trace/condition 诊断。

### 优先级 2：PGH-RHL-lite

理由：它直接针对“纯随机特征缺少语义”的结构性瓶颈，比继续扩大静态 ensemble 权重搜索更可能产生单成员增益。方法完整度包括单原型、多原型、k-means、cosine、RBF 和 sigma 搜索。

必须完成：

- 确定性训练集 prototype bank；
- K=1 cosine；
- fixed/additive budget 与 branch trace；
- K=2/4 k-means；
- global/class/local sigma 及 multiplier 搜索；
- 与 BOA 的 2x2 交互实验。

### 优先级 3：RHL 归一化新方案

理由：旧静态归一化没有涨点，但揭示了 scale、gamma 和类别权重没有被正确拆分。PowerNorm 与 CA-C-RLS 都能直接作用于 BOA/PGH，价值在于校准表示尺度和解析目标，而不是继续重跑 `l2_sqrt`。

必须完成：

- beta 连续路径与 clipping；
- trace-matched gamma；
- train-only class counts；
- old/new、inverse-sqrt、effective-number 权重；
- PowerNorm x CA-C-RLS 2x2；
- 在 BOA/PGH 最佳配置上的 scope 实验。

### 优先级 4：RHL-SE 2.0

理由：oracle 显示存在像素级空间，但现有成员分歧比例低，静态融合没有超过强单模型。它值得完整研究，但单模型表示改造优先。其实现可与优先级 1-3 并行，因为直接使用已有 checkpoint，不占训练主线。

必须完成：

- temperature/calibration；
- margin、entropy、class reliability、consensus、old/new prior；
- rule gate 与 linear gate；
- 图像级 val 数据隔离；
- switch/harm/oracle capture 诊断；
- 同构成员和异构成员分开汇报。

### 优先级 5：Snapshot Analytic Ensemble

理由：它可能提供比 RHL seed 更强的成员多样性，但需要重做 step0 snapshot 记录和多个 step1 run，成本最高，也不是 RHL 单模型机制本身。低优先级表示晚执行，不表示删除方案。

必须完成：

- Snapshot 保存与 manifest；
- 显式 base checkpoint；
- 质量+多样性选择；
- plain RHL 隔离实验；
- 最佳 step1 方法与 SE2 的系统组合。

## 4. 完整执行工作流

## Phase 0：实验协议和配置硬化

### 目标

建立可比较的统一基线，防止脚本默认值和隐藏组合污染后续结果。

### 代码动作

1. 建立 `ExperimentConfig`/结果 manifest，统一记录：global seed、RHL seed、Batch Size、buffer、gamma、init、scale、norm、prototype、sample weight、base checkpoint。
2. 所有 runner 的 run id 编入关键参数；已有输出只读，新实验独立 `SUBPATH`。
3. `eval_rhl_ensemble.py` 增加 member config 兼容性校验。
4. 增加 `--max-train-batches/--max-val-batches/--max-test-batches` 供 smoke test。
5. 固定主基线：Batch32、Buffer8196、gamma1、legacy baseline、norm none、同一 step0 checkpoint。

### 验收

- 默认路径逐元素/指标复现已有强基线；
- checkpoint/config/summary 三处参数一致；
- 任一组合开关都能从 run id 和 manifest 识别；
- shell `bash -n`、Python `py_compile`、单 batch smoke 通过。

## Phase 1A：BOA-RHL 1.5

执行 P1.5 seed、scale/gamma、BUFFER 三组实验。BUFFER 16392 先做资源 smoke，再决定完整训练。每组输出 paired delta、mean/std 和矩阵/特征统计。

阶段结论必须回答：orthogonal 是稳定机制、尺度效应还是随机波动；antithetic 是独立方向不足、规模不可承受还是机制本身无效。

## Phase 1B：RHL-SE 2.0 并行开发

使用已有 seeds 1/2/3，不训练新 RHL 成员：

1. member manifest；
2. image-level val split；
3. temperature 与 reliability stats；
4. rule gate；
5. linear gate；
6. test 冻结评估。

它不阻塞 BOA/PGH 主线。若现有同质成员仍无收益，保留实现，待 Snapshot/PGH 异构成员到位后再验证。

## Phase 2：PGH-RHL-lite

### 2A 原型基础设施

- train-only deterministic loader；
- class count/prototype cache；
- hash/schema/metadata；
- K=1 cosine；
- cache round-trip 和标签泄漏测试。

### 2B 表示完整化

- additive/fixed budget；
- trace-matched branch scale；
- K=2/4 MiniBatchKMeans；
- 空簇、原型使用率、类内距离诊断。

### 2C RBF 完整搜索

- global/class/prototype sigma；
- multiplier `0.25/0.5/1/2/4`；
- val 选择，test 冻结；
- 对 cosine 与 RBF 的稳定性、成本和增益作对照。

## Phase 3：RHL 归一化 v2

### 3A PowerNorm

- beta `0/.25/.5/.75/1`；
- clipping 与 trace；
- fixed gamma vs trace-matched gamma；
- random/prototype/per-branch/joint scope。

### 3B CA-C-RLS

- deterministic train counts；
- all-one compatibility；
- old/new、inverse-sqrt、effective-number；
- cap 与背景权重。

### 3C 内部组合

执行 PowerNorm x CA-C-RLS 2x2，确认两者互补还是冲突。

## Phase 4：单模型组合与交互实验

只有候选模块各自存在稳定证据后，执行：

1. BOA x PGH；
2. 最佳 hidden feature x PowerNorm；
3. 最佳 hidden feature x CA-C-RLS；
4. 必要时做 BOA x PGH x norm 的小型因子设计。

每次至少保留四个角：A off/B off、A on/B off、A off/B on、A on/B on。计算 interaction，不能只报告组合最好结果。

## Phase 5：Snapshot 系统线

1. 在新的 step0 run 中保存候选 Snapshot；
2. 用固定 probe set 计算质量与多样性；
3. 选择 3 个 Snapshot；
4. 每个先用 plain RHL 完成 step1；
5. 再用 Phase 4 的最佳单模型方法；
6. 输出单成员、uniform、class-wise 和 RHL-SE 2.0 对照。

## Phase 6：最终系统与论文证据

构建三层结果表：

- 单机制：BOA、PGH、PowerNorm、CA-C-RLS；
- 组合单模型：最佳互补模块；
- 系统级：Snapshot 成员 + RHL-SE 2.0。

论文表述严格对应证据：单模型机制贡献与系统集成贡献分开，均值提升与稳定性/校准提升分开。

## 5. Codex 实现批次与提交边界

建议按可独立审查的提交组织：

1. `config: harden experiment manifests and compatibility checks`
2. `boa: add seed scale capacity audits and diagnostics`
3. `pgh: add deterministic prototype bank and cosine branch`
4. `pgh: add multi-prototype kmeans and rbf calibration`
5. `rhl: add power normalization and trace diagnostics`
6. `rhl: add class-aware weighted recursive updates`
7. `rhl-se: add calibrated pixel reliability routing`
8. `snapshot: add checkpoint manifests selection and runners`

每个批次均包含：实现、单元测试、runner、配置持久化、smoke test 和对应文档更新。不要把多个方法揉成一个提交，否则后续无法准确 review、bisect 和回滚。

## 7. 组合许可表

| 组合 | 是否允许 | 前提 | 归因方式 |
|---|---|---|---|
| BOA + PGH | 是 | 两者单独完成 | 2x2 interaction |
| BOA + PowerNorm | 是 | scale/gamma 已解耦 | norm on/off paired |
| PGH + PowerNorm | 是 | per-branch trace 已记录 | scope 消融 |

## 8. 最终成功标准

### 方法层

- 至少一个单模型机制在多 seed 下稳定提升 new mIoU，all mIoU 不出现不可接受退化；
- 机制诊断与性能变化方向一致，而非只出现孤立最好点；
- baseline、开关和配置可逐项复现。

### 系统层

- Snapshot 或异构方法确实增加成员分歧且不牺牲成员质量；
- RHL-SE 2.0 能把 oracle 空间的一部分转化为稳定实际收益；
- 组合收益能由因子实验解释。

### 工程层

- 每个方法有专用 runner、独立输出目录和 neutral off switch；
- checkpoint、summary、日志和命令记录一致；
- 单元测试、语法检查和最小运行检查可重复；
- 未完成实验不参与最终结论。

这条路线保留五个方法的完整性，同时让优先级服务于证据质量：先复核已有正信号，再构建语义和尺度机制，随后扩大成员多样性，最后让像素级可靠性路由负责系统融合。
